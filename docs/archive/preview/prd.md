## 1. Product Concept

TripProof는 여행 자료함에 넣은 PDF, 캡처, 메시지, 영수증에 대해 AI와 묻고 답하면서, 사용자가 근거를 보거나 직접 확인한 답변만 카드로 올려 일정 순서의 대시보드와 현장 카드로 정리하는 여행 자료 확인 도구다.

핵심 명사는 하나가 아니다. 이 preview의 핵심은 네 객체가 연결되는 흐름이다.

```text
자료함
→ 자료 기반 채팅 (전체 자료함 대상)
→ 카드 초안 (사람이 확인·편집)
→ 대시보드 (일정 → 카테고리)
→ 현장 카드
```

TripProof는 일반 PDF 챗봇처럼 답변 로그만 쌓지 않는다. 그렇다고 AI가 자료를 읽자마자 대시보드를 자동 완성하지도 않는다.

TripProof의 차별점은 다음이다.

- 사용자는 자료함에 여행 원본을 넣는다.
- AI는 **전체 자료함**을 근거로 답하고, 먼저 확인해볼 후보도 제안한다.
- 답변에는 상태가 붙고, 근거가 있으면 **답변 안에서 바로 펼쳐** 본다.
- 사용자는 답변을 보고 괜찮다고 판단한 것만 카드 초안으로 올린다.
- **AI 근거가 틀릴 수 있으므로, 근거가 부족해도 카드 초안 만들기를 막지 않는다.** 대신 사용자가 직접 값을 채워 올리고, 그 카드는 `직접 확인`으로 표시된다.
- 대시보드는 사람이 올린 카드만 담고, 일정 순서로 정리한다.
- 현장 카드는 대시보드에서 다시 보기로 저장한 정보만 담는다.

## 2. Core Promise

```text
이번 여행 자료를 넣으면,
TripProof가 전체 자료를 근거로 답하고 확인 후보를 제안하며,
사용자가 근거를 보거나 직접 확인해서 괜찮은 답변만 카드로 올리고,
현장에서 다시 볼 정보만 저장하게 해준다.
```

제품이 지켜야 할 약속은 다음이다.

1. 채팅은 실제로 개발되는 핵심 인터랙션이고, **전체 자료함**을 대상으로 한다.
2. 채팅 답변은 항상 상태를 가지고, 근거가 있으면 **답변 안에서 펼쳐** 볼 수 있다.
3. 채팅 답변은 자동으로 카드가 되지 않는다. (사람 확인 게이트)
4. 사용자가 올리기를 눌러야 카드 초안이 되고, 초안을 확정해야 대시보드에 올라간다.
5. **카드로 올릴 수 있는지는 AI 근거 유무가 아니라 사람의 확인으로 결정된다.** 근거가 부족해도 사용자가 직접 채워 올릴 수 있다.
6. 근거 없이 사용자가 채운 카드는 `직접 확인`으로 정직하게 표시된다 — AI 근거 카드와 섞이지 않는다.
7. 민감 정보는 자동 저장하지 않는다. 사용자가 원문에서 확인해 직접 입력해야 한다.
8. 현장 카드는 대시보드에서 저장한 정보만 담는다.

## 3. Target User

TripProof의 사용자는 여행 전에 여러 앱과 파일에 흩어진 자료를 확인해야 하는 개인 여행자다.

이 사용자는 다음 상황에 놓인다.

- 호텔 예약 화면, 바우처 PDF, 호스트 메시지, 영수증이 흩어져 있다.
- 여행 중 또는 출발 직전이라 긴 문서를 다시 읽기 어렵다.
- AI에게 물어보고 싶지만 근거 없는 답은 믿기 어렵다.
- 답변이 괜찮으면 나중에 다시 볼 수 있게 카드로 남기고 싶다.
- AI가 근거를 못 찾아도, 본인이 원문에서 확인한 내용은 직접 채워 남기고 싶다.
- 근거 부족·민감 정보가 근거 있는 것처럼 자동 저장되는 것은 부담스럽다.

## 4. Product Model

### Trip

사용자가 확인하려는 하나의 여행 묶음이다.

예:

```text
오사카 4박 5일
```

### Artifact

사용자가 자료함에 넣은 원본 자료다.

예:

```text
호텔 예약 화면
호스트 체크인 안내
후지산 투어 바우처
렌터카 예약 약관
호텔 도시세 영수증
```

Artifact는 채팅 답변과 카드의 근거가 된다.

### Library Chat

사용자가 **전체 자료함**에 대해 묻는 채팅이다. 특정 자료를 멘션해 범위를 좁힐 수는 있지만, 기본 대상은 자료함 전체다. 문서 하나를 골라서 묻는 PDF 챗봇이 아니다.

예:

```text
호텔에 늦게 도착하면 어떻게 돼?
투어 바우처 프린트해야 해?
렌터카 기름 채워서 반납해야 해?
투어 늦으면 환불돼?
```

Library Chat은 단순 보조 기능이 아니다. 자료를 탐색하고 카드 후보를 발견하는 주 인터랙션이다.

### Chat Answer

자료 기반 채팅의 답변이다.

Chat Answer는 최소한 다음을 가져야 한다.

```text
답변
상태
근거 자료 (있으면)
원문 위치 (있으면)
연결 카테고리
다음 액션
```

근거는 답변 카드 안에서 `근거 N개 펼치기`로 인라인 확인한다. 별도 패널로 떠다니지 않는다. 근거가 없으면 "AI가 못 찾았을 수 있으니 직접 확인이 필요하다"를 명시한다.

다음 액션은 상태와 무관하게 **항상 카드 초안으로 보내기를 허용**한다.

```text
근거 있음:
- 카드 초안으로
- 근거 펼치기

근거 부족 / 자료 충돌:
- 직접 채워 카드 초안 (값 비어 있음)
- 근거 펼치기

민감 정보 (확인 필요):
- 직접 채워 카드 초안
- 원문에서 확인
```

### AI Candidate

자료를 넣었을 때 AI가 먼저 제안하는 확인 후보다. 화면 **오른쪽 rail**에 둔다.

각 후보에서 사용자는 다음을 할 수 있다.

```text
채팅으로 보내기 (질문으로 전환)
카드 초안으로 보내기
근거 펼쳐 보기
```

AI Candidate는 대시보드 카드가 아니다. 사용자가 검토하기 전의 제안이다.

예:

```text
숙소 · 늦은 도착
투어 · 바우처 제시
렌터카 · 연료 정책
결제 · 도시세
```

### Card Draft

Chat Answer 또는 AI Candidate를 사용자가 선택했을 때, **채팅 창 아래**에 뜨는 확인·편집 단계다.

Card Draft는 다음을 보여주고, 편집하게 한다.

```text
카테고리 (선택)
일정 (타임라인 선택)
카드 이름 (편집)
값 (편집)
근거
```

규칙:

- 근거가 없거나 부족하면 경고를 보여주되 막지 않는다.
- 사용자가 값을 직접 채워 올릴 수 있다.
- `대시보드에 올리기`를 누르면 대시보드로 이동한다.

### Dashboard Card

사용자가 카드 초안을 확정해서 대시보드에 올린 카드다. 대시보드 카드는 일정(타임라인) → 카테고리로 묶인다.

```text
숙소
- 체크인 시간: 15:00
- 늦은 도착: 22:00 이후 숙소 연락 필요

투어
- 바우처 제시: 모바일 바우처 가능
```

카드 상태:

```text
근거 있음   AI 근거를 그대로 올린 카드
직접 확인   사용자가 원문 확인 후 직접 채운 카드
현장 저장   현장 카드로 저장한 카드 (overlay)
```

### Evidence (인라인)

답변, 후보, 대시보드 카드가 가리키는 원문 근거다. 별도 패널이 아니라 답변 안에서 펼쳐 본다.

Evidence는 다음을 보여준다.

```text
자료 이름
자료 종류
찾은 위치
짧은 원문
```

근거가 없으면 "이 답변에는 자료 근거가 없다 — AI가 못 찾았을 수 있으니 직접 확인이 필요하다"를 보여준다.

### Field Card

대시보드에서 사용자가 현장에서 다시 보기로 저장한 정보다. 일정별로 크게 본다.

Field Card에는 사용자가 직접 저장하지 않은 정보가 자동으로 들어가면 안 된다.

## 5. Information States

| 상태 | 사용자 문구 | 의미 | 붙는 곳 |
| --- | --- | --- | --- |
| supported | 근거 있음 | 현재 자료 근거로 답할 수 있음 | 답변 · 후보 · 카드 |
| user | 직접 확인 | 사용자가 원문 확인 후 직접 채운 카드 | 카드 |
| needs_review | 확인 필요 | 민감하거나 애매해서 원문 확인이 필요함 | 답변 · 후보 |
| missing | 근거 부족 | 현재 자료로는 AI가 답을 못 찾음 (못 찾았을 뿐일 수 있음) | 답변 · 후보 |
| conflict | 자료 충돌 | 서로 다른 자료가 다른 값을 말함 | 답변 · 후보 |
| saved | 현장 저장 | 사용자가 현장에서 보기로 저장함 | 카드 overlay |

핵심: `needs_review` / `missing` / `conflict`는 **AI가 자료를 얼마나 근거로 댈 수 있었는지**를 말할 뿐, 카드로 올릴 수 있는지를 막는 표시가 아니다. 사람이 직접 확인해 올리면 그 카드는 `user`(직접 확인)가 된다.

나쁜 표현:

```text
AI confidence 0.84
신뢰도 medium
```

좋은 표현:

```text
근거 있음
직접 확인
확인 필요
근거 부족
자료 충돌
현장 저장
```

## 6. Core User Scenario

### 1. 사용자가 자료를 첨부한다

중심 문구:

```text
이번 여행 자료를 첨부하세요.
```

자료함에는 원본이 남는다.

```text
호텔 예약 화면
호스트 메시지
투어 바우처 PDF
렌터카 약관 PDF
도시세 영수증
```

### 2. AI가 추천 후보를 만든다

자료를 넣으면 AI는 먼저 확인해볼 후보를 **오른쪽 rail**에 제안한다. 사용자는 후보를 채팅으로 보내거나, 바로 카드 초안으로 보낼 수 있다.

이 후보는 대시보드가 아니다.

### 3. 사용자가 전체 자료함에 묻는다

사용자는 전체 자료함 기반 채팅으로 직접 묻는다.

예:

```text
호텔에 늦게 도착하면 어떻게 돼?
```

답변:

```text
22:00 이후 도착하면 숙소에 연락해야 합니다.
```

상태: `근거 있음`

근거 (답변 안에서 펼침):

```text
호텔 예약 화면 · 늦은 도착 안내
호스트 메시지 · 셀프 체크인 안내
```

액션:

```text
카드 초안으로
근거 펼치기
```

### 4. 사용자가 답변을 카드 초안으로 올린다

`카드 초안으로`를 누르면 **채팅 아래**에 초안이 생긴다.

```text
카테고리: 숙소
일정: Day 1 · 도착·체크인
카드 이름: 늦은 도착
값: 22:00 이후 숙소 연락 필요
상태: 근거 있음
근거: 호텔 예약 화면 + 호스트 메시지
```

`대시보드에 올리기`를 누르면 대시보드로 이동한다.

### 5. 근거가 없을 때 (사용자가 직접 확인)

근거가 부족해도 막지 않는다.

```text
질문: 투어 늦으면 환불돼?
답변: 현재 자료에서는 지각 시 환불 규정을 찾지 못했습니다.
      (AI가 못 찾았을 수 있으니 직접 확인한 내용이 있으면 채워서 올릴 수 있습니다.)
상태: 근거 부족
```

사용자가 약관을 직접 확인해 초안에 값을 채워 올리면, 대시보드에 `직접 확인` 카드로 올라간다.

### 6. 대시보드에서 관리한다

대시보드는 AI가 자동으로 채운 보드가 아니라, 사용자가 올린 카드의 작업판이다. **일정(타임라인) → 카테고리**로 묶여 있고, 탭으로 전환해 본다.

여기서 사용자는 근거를 다시 보고, 현장에서 다시 볼 정보만 저장한다.

### 7. 현장 카드로 저장한다

현장 카드에는 대시보드에서 저장한 정보만 들어간다. 일정별로 크게 본다.

```text
Day 1 · 도착·체크인
- 체크인 시간: 15:00
- 늦은 도착: 22:00 이후 숙소 연락 필요
```

## 7. UX Structure

TripProof preview는 다음 화면 위계를 가진다.

```text
상단 탭:
- 확인 (채팅)
- 대시보드
- 현장

왼쪽 rail:
- 여행 요약
- 자료함

중앙 (탭별):
- 확인: 자료함 채팅 + 그 아래 카드 초안
- 대시보드: 일정 → 카테고리 카드
- 현장: 현장 카드

오른쪽 rail:
- AI 추천 후보 (채팅으로 / 카드 초안 / 근거)
```

대시보드를 탭으로 분리한 이유는, 채팅·초안·대시보드·현장을 한 화면에 모두 펼치면 보기 어렵기 때문이다.

### 자료함

역할:

- 원본 자료를 보존한다.
- 채팅과 근거의 출처를 보여준다.
- 새 자료를 첨부하게 한다.

### AI 추천 후보 (오른쪽)

역할:

- AI가 먼저 확인할 만한 정보를 제안한다.
- 후보를 채팅으로 보내거나 카드 초안으로 보낼 수 있게 한다.
- 확정 대시보드처럼 보이면 안 된다.

### 자료함 채팅 (확인 탭)

역할:

- 사용자가 전체 자료에 직접 묻는다.
- 답변에 상태를 붙이고, 근거는 답변 안에서 펼쳐 본다.
- 답변을 카드 초안으로 올리는 입구가 된다.

### 카드 초안 (채팅 아래)

역할:

- 답변이나 후보가 카드가 되기 전, 사람이 내용을 확인·편집하는 단계다.
- 카테고리·일정·이름·값을 보여주고 편집하게 한다.
- 근거가 부족해도 막지 않고, 직접 채워 올릴 수 있게 한다.
- 확정하면 대시보드로 이동한다.

### 대시보드 (대시보드 탭)

역할:

- 사용자가 확정한 카드만 모인다.
- 일정(타임라인) → 카테고리로 묶어 보여준다.
- AI 근거 카드(`근거 있음`)와 직접 채운 카드(`직접 확인`)를 구분한다.
- 여기서 현장 카드로 저장한다.

### 현장 카드 (현장 탭)

역할:

- 대시보드에서 저장한 정보만 보여준다.
- 일정별로 크게, 현장에서 빠르게 다시 볼 정보의 도착점이다.

## 8. Chat And Card Integration

채팅은 낮은 보조 기능이 아니다. 다만 채팅 로그 자체가 최종 산출물도 아니다.

역할은 다음처럼 나뉜다.

```text
자료함:
원본과 질문 대상

채팅:
전체 자료를 탐색하고 답변을 받는 핵심 인터랙션

답변:
상태와 (있으면) 근거가 붙은 카드 후보

카드 초안:
사용자가 답변을 구조화하고 편집해서 검토하는 중간 단계

대시보드:
사용자가 올린 카드의 작업판 (일정 → 카테고리)

현장 카드:
사용자가 다시 보기로 저장한 결과물
```

채팅이 제품을 흐리게 만들지 않으려면 다음 규칙이 필요하다.

1. 질문에는 전체 자료를 근거로 답한다.
2. 답변은 근거 없이 확정형으로 끝나면 안 되고, 상태를 가진다.
3. 근거가 있으면 답변 안에서 펼쳐 본다.
4. 답변은 자동으로 카드가 되지 않는다.
5. 사용자가 `카드 초안으로`를 눌러야 초안이 되고, 초안을 확정해야 대시보드에 올라간다.
6. **근거 부족·충돌이어도 초안을 막지 않는다.** AI가 못 찾았을 수 있으므로, 사용자가 직접 값을 채워 올릴 수 있다.
7. 직접 채운 카드는 `직접 확인`으로 표시해 AI 근거 카드와 구분한다.
8. 민감 정보는 자동 저장하지 않고, 원문 확인 후 직접 입력하게 한다.

## 9. MVP Scope

포함:

- 이번 여행 자료 첨부
- 자료함
- 오른쪽 rail의 AI 추천 후보 (채팅으로 / 카드 초안)
- 전체 자료함 기반 채팅
- 답변의 상태 표시와 근거 인라인 펼침
- 답변·후보에서 채팅 아래 카드 초안 만들기 (이름·값 편집)
- 근거 부족·민감 답변도 직접 채워 올리기 (`직접 확인` 카드)
- 대시보드에 카드 올리기 (일정 → 카테고리, 탭 전환)
- 대시보드에서 현장 카드 저장

제외:

- 예약, 결제, 환불 처리
- 항공편 실시간 상태 조회
- 범용 문서 금고
- 문서 하나만 고르는 PDF 챗봇 모드
- 사용자 확인 없는 자동 카드 생성
- 사용자 확인 없는 현장 카드 저장

## 10. Success Criteria

TripProof의 UX가 성공하려면 사용자가 다음을 이해할 수 있어야 한다.

1. 자료함에 자료를 넣고, **전체 자료**에 물어본다.
2. 채팅 답변에는 상태가 붙고, 근거는 답변 안에서 펼쳐 본다.
3. 추천 후보는 오른쪽에서 채팅으로 보내거나 카드 초안으로 보낸다.
4. 답변을 보고 괜찮다고 판단하면 채팅 아래 초안에서 확인해 올린다.
5. **근거가 부족해도 직접 채워 올릴 수 있고, 그 카드는 `직접 확인`으로 구분된다.**
6. 대시보드에는 사람이 올린 카드만 일정 순서로 들어간다.
7. 현장 카드에는 대시보드에서 저장한 정보만 들어간다.
8. 사람 확인 없이 자동 저장되는 카드는 없다.

실패 신호:

- 채팅이 문서 하나를 고르는 PDF 챗봇처럼 보인다.
- 근거가 결정과 분리된 채 따로 떠다닌다.
- 근거 부족이라고 카드로 못 올린다 (AI 판단이 사용자를 막는다).
- 직접 확인 카드와 AI 근거 카드가 구분되지 않는다.
- AI가 대시보드를 자동 완성한 것처럼 보인다.
- 대시보드가 한 화면에 다 안 들어와 보기 어렵다.

TripProof는 여행 자료에 대해 AI와 묻고 답하면서, 근거 있는 답변이든 직접 확인한 내용이든 사용자가 카드로 올려 일정 순서의 대시보드와 현장 카드로 정리하는 도구다.
