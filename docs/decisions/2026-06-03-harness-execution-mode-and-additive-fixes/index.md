# 2026-06-03 - 하네스 문서 역할 재정렬

## 폴더 구성

- 기존 `docs/decisions/2026-06-03-harness-execution-mode-and-additive-fixes.md`는 초기 결정 스냅샷으로 그대로 둔다.
- `index.md`: 외부 하네스 조사와 후속 대화를 반영한 확장 재진입본.
- `raw.md`: 이 결정이 생긴 대화, 외부 하네스 조사, 보조 에이전트 검토에서 나온 생생한 판단 재료.

`raw.md`는 현재 실행 기준이 아니다. 다음 세션이 이 결정을 왜 했는지 이해하기 위한 배경 자료다.

## 맥락

TripProof는 여행자가 받은 예약·안내 자료에서 지금 행동에 필요한 정보를 확인하는 제품이다. 이 repo에는 제품 기준, 기능 기준 문서, 로드맵, 작업 기록, 결정 기록, 보조 에이전트 정의 같은 문서가 함께 있다.

여기서 하네스는 AI와 개발자가 작업할 때 참고하는 운영 문서 전체를 뜻한다. 하네스 자체가 문제는 아니다. 하네스는 제품 기준을 보존하고, 다음 작업자가 빠르게 맥락을 잡게 하며, 아직 구현되지 않은 것과 이미 결정된 것을 구분하게 해준다.

이번 문제는 하네스가 너무 많은 역할을 동시에 맡으면서 생겼다. 일부 문서는 제품 기준, 과거 작업 기록, 후보 목록, 구현 순서, 검증 아이디어, AI 운영 규칙을 한꺼번에 담고 있었다. 그 결과 다음 AI나 작업자가 문서를 읽을 때 "참고할 배경"과 "지금 따라야 할 명령"을 구분하기 어려웠다.

특히 아래 흐름이 반복됐다.

- 로드맵과 작업 기록의 후보가 현재 작업 대기열처럼 읽힘.
- 기능 기준 문서의 통과 기준이 구현 순서나 선행 관문처럼 읽힘.
- 과거 분석과 보조 에이전트 보고가 현재 판단을 대체하는 근거처럼 읽힘.
- 문서를 고쳐야 할 때 삭제·교정·격리 대신 보완문을 덧대어 문제 대상을 계속 살려둠.
- 항상 켜지는 `CLAUDE.md` / `AGENTS.md`에 운영 규칙을 계속 추가하려는 압력이 생김.

따라서 이번 결정은 하네스를 없애는 결정이 아니라, 문서별 역할을 낮추고 분리하는 결정이다.

## 결정

TripProof 문서는 제품 기준을 보존하되, 현재 실행 명령처럼 읽히지 않게 재정렬한다.

### 항상 켜지는 문서

`CLAUDE.md` / `AGENTS.md`는 AI가 항상 읽는 얇은 실행 경계만 둔다.

- 하네스와 문서는 실행을 돕는 기준이지 현재 작업 승인권이 아니다.
- 큰 변경은 사용자에게 먼저 알린다.
- 이미 합의된 작업 단위 안의 작은 구현 판단은 멈춰 묻기보다 진행하고 결과로 설명한다.
- 보조 에이전트는 보고 전용 판단 재료다.
- 사용자가 "두고", "멈춰", "보류"라고 말하면 파일 수정과 추가 실행을 멈춘다.

이 파일에는 제품 spec, 과거 분석, 자세한 문서 운영 규칙을 넣지 않는다.

### 자연어 문서와 실제 강제 장치

자연어 문서는 작업자가 판단할 기준과 맥락을 남길 뿐이다. 실제 강제는 test, CI, hook, permission 같은 별도 장치가 생겼을 때 그 장치의 위치에서 명시한다.

TripProof의 문제는 강제 장치가 너무 강해서 생긴 것이 아니었다. 실제 강제 장치가 거의 없는 상태에서 roadmap, spec, work-log의 자연어 문장이 작업 대기열, 선행 관문, 승인권처럼 읽힌 것이 문제였다.

### 문서 역할 안내

`README.md`는 repo 문서의 역할을 안내한다.

- `docs/product-model.md`: 제품 어휘·상태·흐름 기준.
- `docs/prd.md`: 제품 요구사항과 경계.
- `docs/specs/`: 제품 동작 통과 기준.
- `docs/roadmap/`: 후보 메뉴와 출발점 기록.
- `docs/work-log.md`: 과거 작업의 얇은 재진입 기록.
- `docs/decisions/`: 이후 구현 방향에 영향을 주는 선택 기록.
- `docs/archive/`: 동결 스냅샷과 참조 보관소.

README는 문서 운영 규칙표가 아니라, 각 문서를 어떤 권한으로 읽을지 알려주는 지도 역할만 한다.

### Roadmap

`docs/roadmap/v1.md`는 일정표나 단계 계약이 아니라 후보 메뉴로 낮춘다.

- "0단계, 1단계, 2단계"처럼 차례를 강하게 만드는 표현을 후보형으로 바꾼다.
- 구조 정비, MVP 흐름, 후속 확장은 현재 명령이 아니라 열 수 있는 후보로 둔다.
- 실제 선택은 사용자 요청과 현재 코드 상태에서 닫는다.

### Specs

`docs/specs/`는 구현 순서가 아니라 제품 동작 통과 기준을 남긴다.

- `docs/specs/accommodation-checkin.md`의 P0 case는 현재 작업 대기열이 아니라 관찰 기준이다.
- 구현 순서와 "다음 검증" 목록은 제거한다.
- 구현 냄새 신호는 선행 gate가 아니라 결과 검토 때 볼 위험 신호다.
- 보류 질문은 현재 작업을 막지 않는다.

`docs/specs/00-spec-driven-development.md`도 기능 기준 문서를 쓰는 기준이지 절차 관문이 아니다. 최소 루프는 고정 순서가 아니라 작업 설명을 작게 잡기 위한 감각으로 둔다.

### Product Model

`docs/product-model.md`는 제품 어휘와 상태의 기준으로 둔다. 타입 확장 후보나 코드 드리프트는 현재 구현 명령이 아니라, 실제 slice에서 필요해질 때 다시 판단할 항목이다.

### Work Log

`docs/work-log.md`는 과거 작업 기록이다. 여기에 남은 후보는 다음 작업 지시가 아니다. "다음"처럼 읽히는 표현은 "남은 관찰"에 가깝게 낮춘다.

## 외부 하네스 조사에서 가져온 판단 후보

외부 하네스 레퍼런스는 권위로 적용하지 않는다. 현재 문제를 비춰 보는 calibration으로만 쓴다. 조사에서 얻은 후보는 아래처럼 반영한다.

### 1. 항상 켜지는 문서는 orientation이다

OpenAI Codex의 `AGENTS.md`, AGENTS.md convention, GitHub Copilot repository instruction은 repo-wide 문서를 작업 전에 읽는 맥락 자료로 둔다. 여기서 가져올 점은 "항상 켜지는 문서는 현재 작업 명령서가 아니다"라는 권한 분리다.

TripProof에는 이미 반영했다. `CLAUDE.md` / `AGENTS.md`는 얇은 실행 경계로 두고, 제품 spec이나 과거 분석을 넣지 않는다.

### 2. 항상 켜지는 instruction은 짧고 넓게 둔다

Cursor, Continue, Cline, GitHub Copilot 계열 문서는 repo-wide instruction과 path-specific 또는 on-demand rule을 나눈다. 여기서 가져올 점은 세부 규칙을 root에 계속 쌓지 않는 것이다.

TripProof에는 root 문서를 더 늘리는 대신 문서별 역할을 낮추는 방식으로 반영했다. 아직은 `.clinerules`, `.continue/rules` 같은 새 구조를 만들지 않는다. 영역별 지침이 실제로 커질 때만 좁은 scope 문서를 다시 판단한다.

### 3. 자연어 instruction과 enforcement를 분리한다

Claude Code hooks, OpenHands hooks, CI/test 관행은 실제로 막거나 실행되는 계층과 자연어 문서 계층을 분리한다. 이 항목은 TripProof 문제와 직접 연결된다.

TripProof에서는 실제 강제 장치가 거의 없었기 때문에 자연어 문서가 강제 역할까지 떠안았다. 그래서 "문서가 많음"보다 "문서가 주는 실행 권한"이 문제였다. 이번 결정은 문서를 없애는 것이 아니라, 문서가 실제 gate인 척하지 않게 역할을 낮추는 것이다.

### 4. 조사·계획과 실행 권한을 분리한다

Continue Plan Mode, Aider의 read-only convention, Claude subagent 같은 사례는 조사와 실행 권한을 분리한다. 여기서 가져올 점은 보조 에이전트와 조사 결과를 승인권으로 승격하지 않는 것이다.

TripProof의 `doc-boundary-reviewer`와 `anchor-expression-reviewer`는 report-only로만 둔다. 호출 여부가 작업을 막지 않고, 결과도 main 판단과 사용자 의도 안에서만 쓰인다.

### 5. eval과 spec은 product behavior를 관찰한다

OpenAI eval, BDD, 일반 CI/test 관행에서 가져올 점은 eval과 spec이 제품 동작을 관찰한다는 것이다. spec과 fixture가 제품 구현 순서나 제품의 상위 권한이 되면 안 된다.

TripProof에는 product-first 원칙으로 반영했다. `docs/specs/`는 product behavior acceptance이고, `eval/`은 product code를 호출해 관찰한다. product가 eval fixture, run artifact, metric output에 의존하지 않는다.

## 덧대기식 보완 경계

문서를 고칠 때 먼저 대상의 상태를 판단한다.

- **삭제**: 이제 필요 없거나 폐기된 대상.
- **교정**: 대상은 필요하지만 표현, 출처, 의미가 틀린 경우.
- **격리**: 과거 기록으로는 필요하지만 현재 작업 기준이면 안 되는 경우.
- **보완**: 대상 자체는 여전히 유효하고 부족한 설명만 채우면 되는 경우.

문제는 보완 자체가 아니다. 삭제·교정·격리해야 할 대상을 그대로 둔 채 "이건 기준 아님", "하지 말 것", "주의" 같은 문장을 덧대면, 그 대상은 계속 현재 문맥에 남는다. 다음 AI는 그 대상을 다시 후보나 근거처럼 읽을 수 있다.

이번 결정에서 이 흐름을 "덧대기식 보완"으로 부른다. 개발로 비유하면 잘못된 값을 제거하거나 구조를 고치는 대신 예외 분기를 계속 추가하는 것과 비슷하다.

## 보조 에이전트

이번 작업에서 두 repo-local 보조 에이전트를 둔다. 둘 다 사람이 판단하기 전에 위험을 보고하는 도구일 뿐이다.

- `doc-boundary-reviewer`: 문서가 현재 실행 명령, 절차, 작업 대기열, 관문처럼 읽히는지 점검한다.
- `anchor-expression-reviewer`: 삭제·교정·격리해야 할 대상을 보완문으로 덧대어 재호출 가능하게 남겼는지 점검한다.

두 에이전트 모두 보고 전용이다. 직접 파일을 수정하지 않고, 승인/반려 권한도 없으며, 필수 관문이 아니다. 호출 여부가 작업을 막지 않는다.

## 기각 또는 보류

- 하네스와 기능 기준 중심 작업 자체를 폐기하는 결론은 기각한다. 제품 정직성, 근거 상태, 통과 기준은 남겨야 한다.
- "생각을 닫고 실행만 하자"는 결론도 기각한다. 해결되지 않은 문제에서는 생각이 열릴 수 있고, 이 repo에는 제품 기준을 보존하는 사고가 필요하다.
- 모든 문서 수정 전에 에이전트를 반드시 실행하는 방식은 기각한다. 그것도 새 관문이 된다.
- 과거 대화 원문, 개인 AI 사용 노트, 보조 에이전트 보고를 현재 작업 기준으로 승격하는 방식은 기각한다.
- reset 전 stash의 분석 문서를 그대로 복원하는 것은 보류한다. 현재 문서에 필요한 판단만 흡수한다.
- 외부 하네스 레퍼런스를 권위처럼 적용하는 것은 기각한다. 다만 이번 조사에서 확인한 5개 패턴은 위의 판단 후보로 흡수한다.

## 검증

- 이전 작업 흐름은 문서가 많다는 사실만으로는 설명되지 않았다. 다른 프로젝트에는 더 많은 하네스 문서가 있었지만 작업이 닫힌 사례가 있었다.
- TripProof에서 문제를 만든 쪽은 문서량보다 문서가 주는 실행 권한이었다.
- 현재 수정은 root 지침, README, roadmap, spec, product model, work-log의 역할을 각각 낮추고 분리했다.
- 보조 에이전트는 `.claude/agents/`와 `.codex/agents/`에 이 repo 전용으로만 추가했다. 전역 에이전트나 전역 스킬로 승격하지 않았다.

## 이번 결정 밖

- 문서 수정이 다시 필요하면 "보완"부터 하지 말고 대상 상태를 먼저 본다: 삭제, 교정, 격리, 보완.
- 작은 구현 판단은 코드와 가까운 곳에서 닫고, 큰 변경은 사용자에게 먼저 알린다.
- 에이전트 결과는 판단 재료로만 쓰고, main 대화에서 사용자 의도와 현재 코드 상태를 함께 보며 닫는다.
- path-scoped rule, 별도 hook, CI/test gate는 지금 만들지 않는다. 실제 product slice에서 필요해질 때 위치와 실패 조건을 좁게 잡아 다시 판단한다.
