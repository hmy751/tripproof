# Raw Notes - 큰 feature spec과 슬라이스 빌드 모델

이 파일은 `index.md` 결정의 배경 재료이며, 거쳐온 고민과 트레이드오프를 보존하기 위한 자료다. 현재 실행 기준이나 작업 대기열이 아니다. 결론은 `index.md`에 있고, 여기 후보·표현을 현재 작업으로 쓰려면 `index.md`와 현재 코드 상태로 다시 판단한다.

2026-06-05 후속 정리에서 현역 어휘는 `feature`(spec 단위), `slice`(빌드 단계), `scenario`(사례), `acceptance criteria/AC`로 맞췄다. 아래의 "시나리오", "슬라이스" 표현은 당시 고민의 흔적일 수 있으므로 현재 기준은 `index.md`를 따른다.

## 왜 raw가 필요한가

같은 결론에 여러 차례 도달했지만, 기록 위치와 권한이 분명하지 않으면 같은 논의가 반복됐다. "어떤 고민을 했고 무엇을 왜 버렸나"를 남겨야 같은 판단을 다시 반복하지 않는다. 결정 자체보다 **거쳐온 트레이드오프**가 이 파일의 핵심이다.

## 거쳐온 고민 사례와 트레이드오프

### 1. 표면 문제 vs 실제 문제

- 표면: "시나리오가 크면 하위 spec이 안 쪼개진다 / 문서만 늘어난다."
- 실제: 분해는 진짜 필요한데, 그 분해 기록이 product 기준과 같은 권위를 가지면 작업지시서가 된다. 문제는 "분해의 존재"가 아니라 "분해 기록의 권한"이었다.

### 2. spec을 버릴까 vs 더 만들까 (양 극단, 둘 다 기각)

- 한쪽 극 "기록 없이 코드만 작성": 빠르지만, 다 만든 뒤 기록이 흩어져 한 흐름으로 다시 엮기 어렵다. 개발 속도만 보고 기록을 깎는 쪽.
- 다른 쪽 극 "큰 feature마다 새 spec 파일": 기록은 남지만 기존 product 문서와 중복(drift) + 문서 비대 + gate화.
- 트레이드오프 끝에 둘 다 기각. 통과 기준은 이미 기존 문서가 소유하고, 분해는 진행 메모 + commit으로.

### 3. 개발 병목 — 부품 목업이 흐름을 막는다

- 과거 실패 감각: 큰 feature를 한 번에 만들려다 부품마다 임시 목업을 만들고, 그 목업을 다듬느라 정작 end-to-end 흐름이 늦어질 수 있다.
- 해소 방향: 부품을 가로로 다 만들지 말고, **한 필드를 실데이터로 처음부터 끝까지** 한 번 통과시킨다(나머지는 stub). 부품 목업을 따로 만드는 순간이 곧 병목이므로 그걸 금지선으로 둔다.
- 트레이드오프: 첫 화면이 비어 보인다(필드 하나만 실데이터). 그러나 그 "비어 보임"이 정직한 진행 상태고, 완료를 컴포넌트가 아니라 화면 동작으로 재면 목업으로 빠질 길이 막힌다.

### 4. 분해를 어디에 적나 (spec 안 vs 별도 파일 vs 안 적음)

- spec 안: 가까이서 보여 좋지만, 통과 기준과 섞이면 작업지시서가 된다.
- 별도 work-map 파일: 분리는 되지만 영구 tracked 파일에 번호 매긴 목록을 두면 queue/gate가 재발한다. "두 현역 기준 문서 금지" 결정과도 충돌.
- 안 적음(습관으로만): 문서 비대화는 피할 수 있지만 같은 논의가 반복된다.
- 결론: spec 안에 두되 "진행 메모(통과 기준 아님)"라는 가드를 붙인다. 별도 work-map 파일은 만들지 않는다.

### 5. 명시 필요 vs 작업 순서표 금지

- 명시하면 반복 논의를 줄이고 나중에 흐름을 따라 읽기 좋다. 그러나 슬라이스 목록을 명시하는 것은 "순서대로 구현하라"는 작업 순서표처럼 보일 수 있다.
- 해소: 명시 대상을 두 층으로 나눈다. 재사용되는 **방법 원칙**(펼치기·세로 슬라이스·목업 금지)은 작업 감각이라 안전하게 명시할 수 있고, **per-feature 슬라이스 목록**은 진행 메모/commit으로 일회용. 둘을 같은 권위·같은 수명에 두지 않는다.

### 6. descriptive vs prescriptive (충돌을 푼 관점 전환)

- spec을 "미리 써서 막는 gate(prescriptive)"로 보면 분해를 spec에 넣는 순간 게이트가 된다.
- spec을 "개발하며 생기는 기록(descriptive)"으로 보면 같은 분해 목록이 재진입 메모가 된다.
- 이 관점 전환이 "기록하며 제어"와 "gate 회피"가 양립 가능함을 보여줬다. 결정의 1번이 여기서 나왔다.

### 7. worktree 병렬 장치를 살릴까

- 큰 spec을 worktree로 병렬 분해하는 장치는 여러 사람·대규모엔 의미가 있다. 그러나 현재 단계에서는 end-to-end product path가 아직 얇아 과설계였고, 조기 분해를 유도했다.
- 결론: 현 단계에서는 사용하지 않는다. 유용한 핵심(구현면 펼치기·slice·stub 감각)만 로컬 작업 루프로 흡수한다.

### 8. 용어 혼선

- "slice"가 spec 레벨(큰 것)과 빌드 단계(작은 것) 양쪽에 쓰여 충돌했다. 외부 방법론 용어(feature, capability, walking skeleton, track 등)를 본래 작업 어휘와 섞어 매 논의가 헷갈렸다.
- 후속 논의에서 `feature`는 spec이 다루는 기능 단위로 가져오고, 기존 "시나리오"는 큰 단위 이름에서 내렸다. BDD 쪽 표준 감각에 맞춰 `scenario`는 feature 안에서 AC를 확인할 구체 사례로 쓴다.
- 정리: feature(spec 단위) / slice(순서대로 짓는 빌드 단계) / scenario(사례) / acceptance criteria 또는 AC(통과 조건) / 부품(구현 조각). `capability`, `track`, work-map, walking skeleton 같은 외부 어휘와 무거운 파이프라인은 들이지 않는다.

## 출처와 확인된 사실 (당시 코드 기준)

- 추출 baseline은 숙소 체크인 자료에서 체크인 시작 시간(근거 있음), 늦은 도착 조건(신뢰도 낮아 확인 필요), 출입 코드(민감, 값 없음)를 deterministic하게 만들고 test가 통과한다.
- 정규화 코드는 진짜 근거 상태 로직을 갖고 shared 계약을 쓴다. 서버도 shared 계약을 쓴다.
- 그러나 클라이언트는 아직 shared/server/AI 흐름과 연결되어 있지 않고, 질문 입력은 하드코딩 문자열만 반환한다. 필요한 것은 새 운반 장치가 아니라 **이미 있는 두 반쪽을 잇는 코드 경로**였다.
- feature 단위 통과 기준은 사라진 적이 없다 — 기존 product 문서와 baseline 코드/test가 이미 나눠 소유한다.

## 기각·보류된 후보 (현재 후보 아님)

- 별도 work-map / execution-map 파일 — 기각.
- 분해를 product spec 본문에 통과 기준처럼 심기 — 기각.
- worktree 병렬 분해 장치 — 현 단계에서 보류.
- feature마다 AC를 재서술하는 새 spec 파일 — 보류.

## 과정에서 드러난 재발 위험

- 논의가 문서 구조 개선으로 너무 빨리 수렴하면 product 실행 기준이 흐려질 수 있다.
- 추가 검토에서도 별도 실행 지도가 이전 실패를 반복할 위험이 확인됐다.
- 교훈: 막혔을 때 기본값은 더 작은 쪽(코드 경로, 진행 메모, 태그 commit)이며, 새 문서층은 실제로 그 문제가 코드로 나타날 때만 다시 판단한다.

## 후속 관찰

- 첫 슬라이스를 아직 화면에서 돌리지 않아, 이 모델은 product로 검증되지 않았다.
- 슬라이스 목록이 실전에서 "진행 메모"로 유지될지, 다시 "queue"로 굳을지는 관찰이 필요하다.
- 첫 slice 구현(client ↔ shared 계약 연결)에서 이 모델이 실제로 유지되는지 확인한다.
