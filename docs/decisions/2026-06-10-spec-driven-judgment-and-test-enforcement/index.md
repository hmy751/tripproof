# 2026-06-10 - spec-driven 하네스: 문서 게이트에서 판단·테스트 강제로

## 폴더 구성

- `index.md`: 결론과 현재 판단.
- `raw.md`: 이 결정에 이른 출발 직감, 거쳐온 교정, 트레이드오프.

`raw.md`는 현재 실행 기준이 아니다. 다음 세션은 `index.md`를 먼저 읽고, 왜 이런 결정을 했는지 필요할 때만 `raw.md`를 본다.

## 맥락

배경: decisions, work-log, git 이력에서 보이듯 spec-driven 운영이 제품 구현을 무겁게 만든다는 감각이 쌓였다. `2026-06-03-light-spec-driven-loop`와 `2026-06-05-slice-build-model`로 루프는 이미 가볍게 낮췄지만 한 가지 마찰이 남았다. 하네스에 자동 강제장치(test gate, CI, hook)가 0이라 prose(SKILL.md, README)가 유일한 강제 수단이었고, 문제가 보일 때마다 문서에 보완문을 더하는 방식으로 커졌다. prose가 유일 강제장치일 때 문서는 가이드가 아니라 통과 게이트로 굳고 비대해진다.

목적: "spec-driven이 구현을 무겁게 했다"의 실체를 분량 문제가 아니라 강제 방식 문제로 본다. 그래서 (1) 강제력을 문서에서 제품행동 테스트로 옮기고, (2) 문서는 규칙을 나열하는 대신 구현 직전 판단을 돕는 쪽으로 되돌린다. 단 기존 누적을 전부 불필요로 보지 않는다 — 최근에 박힌 guard 일부는 실제로 막 작동하기 시작했다.

이 결정은 light-spec-driven-loop와 slice-build-model의 product-first·문서 비대화 방지 기준을 그대로 잇는다.

## 근거

- prose는 코드를 짜는 그 순간의 attention을 바꾸지 못한다. SKILL.md의 Runtime Smells가 `hard-coded answer 금지`, `deterministic path를 LLM처럼 표현 금지`를 말로 다 금지했는데도, 같은 종류의 코드 내부 규칙 처리가 한동안 라이브 경로에 남았다. 금지문을 더 추가하는 것은 같은 실패를 반복한다.
- 약한 구현 판단은 지식 부족이 아니라, "이번 작업을 가장 빨리 통과시키는 코드 조각"에 attend한 상태에서 코드 내부 규칙(키워드·패턴·고정 phrase)이 가장 그럴듯한 다음 선택으로 떠오르기 때문에 나온다. 그래서 처방은 증상 탐지(lint/grep)가 아니라 결정 시점의 판단 사전분포를 바꾸는 것이다.
- 이 repo엔 이미 작동하는 교정 루프가 있다. 경계 관찰을 기록한 implementation-note가 남은 뒤 며칠 안에 실제 코드 경로(local 후보 생성 → 실제 provider 호출)가 그 drift를 닫은 인과가 확인됐다. 새 장치를 발명하기보다 이 사례를 결정 컨텍스트로 끌어오면 같은 미끄러짐이 비슷한 장면에서 애초에 덜 떠오른다.
- 누적된 prose는 시점(vintage)으로 갈라야 한다. 초기 churn 흔적과 최근 막 작동하기 시작한 자기교정 guard를 한 덩어리로 묶어 삭제하면 자라는 학습 루프를 함께 자른다.

## 결정

spec-driven 하네스의 강제 방식을 prose에서 (1) 결정-시점 판단 질문과 (2) 제품행동 테스트로 옮기고, 문서는 판단을 돕는 reference로 둔다.

1. **제품 흐름 경계를 행위자 질문으로 전환.** `제품 흐름 경계` 섹션에서 부정 금지문 두 줄(LLM 불확실성을 hard-coded answer로 대체 금지 / fallback을 실제 LLM 판단 경로처럼 표현 금지)을 한 줄의 결정-시점 질문으로 치환했다: 값/판정을 만드는 실제 행위자가 LLM 판단인지 코드 내부 규칙인지, 자료가 바뀌면 출력이 바뀌는지가 흐려지면 멈춰서 본다.
2. **사례 calibration 꼬리.** 이 판단이 애매할 때만 추상 규칙이 아니라 `docs/implementation-notes/`에서 같은 미끄러짐을 기록한 비슷한 사례를 찾아 참고하게 했다(선행 필독 아님). 두 노트를 대표 예시로 단다.
3. **발견성 포인터.** 구현 직전 단계(`우선순위` 6)에서 이 행위자 질문을 한 줄로 가리키게 해 결정 흐름에서 닿게 했다.
4. **제품행동 불변식을 테스트로 못박음.** 자료가 바뀌면 답이 바뀌는지(메타모픽), `근거 있음`이면 근거가 source의 실제 부분 문자열인지(substring grounding), 질문 경로가 답변 composer 계약을 부르는지(라우팅)를 테스트로 고정했다.
5. **미사용 추출 경로 제거.** 채팅 앞면에서 빠진 뒤 호출자가 0이던 체크인 fact 추출/proposer 경로를 삭제했다.
6. **문서 자체는 추가 축소하지 않음.** 테스트가 생긴 뒤 SKILL.md에서 줄일 prose를 다시 점검한 결과, 코드/테스트로 강제되는 두 줄(메타모픽·substring)도 단일 답변 경로만 덮고 문장은 전 제품 흐름 invariant라 자르면 미커버 경로의 가드를 잃었다. 나머지는 테스트로 못 잡는 판단 원칙이었다. 줄일 분량이 사실상 없다는 것은 문서의 무거움이 분량이 아니라 강제 방식이었음을 확인해 준다.

또 skill 발동 기준을 좁혀, spec/AC/product contract가 현재 판단 기준일 때만 켜지고 일반 구현 설명·코드 탐색에는 발동하지 않도록 정리했다.

## 기각 또는 보류

- **prose 전량 삭제(SKILL.md 통째 축소) — 기각.** 대부분 테스트로 옮길 수 없는 판단 원칙이고, 최근 작동하는 guard까지 함께 죽는다.
- **증상 탐지 도구(코드 규칙 우회·하드코딩·목업을 잡는 lint, grep, CI 룰) 신설 — 기각.** 증상을 잡으면 모델은 걸리지 않는 다른 우회를 찾을 뿐 판단이 자라지 않는다. 코드 규칙 우회는 증상의 대표 예시지 타깃이 아니다.
- **행위자를 직접 감시하는 spy 테스트(LLM이 판정을 정하는지 확인) — 보류/폐기.** 라이브 경로는 이미 LLM이 evidence_state를 정해 trivially green이라 새 강제력을 못 만든다. 대신 우회의 결과(근거 없는 `근거 있음`, 끊긴 경로)를 substring 불변식과 라우팅 테스트로 방어한다.
- **메타모픽을 전역 개발 규칙으로 승격 — 보류.** 좋은 기법이지만 지금은 이 repo의 제품행동 테스트로만 둔다.
- **행위자 질문·사례 앵커를 매번 통과해야 하는 선행 게이트로 굳히기 — 기각.** 그 순간 doc-as-gate로 회귀한다. 질문은 답하면 끝나는 결정 행위, 앵커는 calibration sample로만 두고, 작은 작업엔 생략(이유 한 줄)을 허용한다.

## 검증

이 전환이 작동하는지, 과해지는지는 다음으로 관찰한다.

기대한 작동:

- 새 코드 구현 시 모델이 결정 직전 "이 값을 만드는 실제 행위자가 누구인가 / 자료가 바뀌면 출력이 바뀌는가"에 attend해, 코드 규칙 우회가 비슷한 장면에서 덜 떠오른다.
- 제품행동 불변식이 문서 문장이 아니라 테스트로 지켜진다.
- 문제가 생기면 첫 반응이 보완문 추가가 아니라 직전 기준 복구 검토다.

과해짐 신호(보이면 되돌린다):

- 행위자 질문이나 사례 앵커가 구현 전 매번 거쳐야 하는 선행 게이트로 읽힌다.
- 테스트가 product behavior가 아니라 특정 fixture 문구·seed 값 맞추기로 좁아진다.
- 최근 guard를 초기 churn 흔적과 한 덩어리로 묶어 일괄 삭제하려 한다.
- 문서가 다시 보완문 누적으로 비대해진다.

## 이번 결정 밖

- CI/hook 같은 자동 게이트 도입 시점. 현재는 `npm test` 수동 실행만 있고 자동 게이트는 없다.
- 메타모픽·substring 불변식 테스트를 다른 도메인 경로(ingestion, card 승격 등)로 확장하는 일.
- `docs/specs/README.md`의 동일한 금지문을 결정-시점 질문/사례로 정렬할지, 아니면 product acceptance 계약 기준으로 prose를 유지할지.
- 이 변경 묶음을 main에 통합하는 시점.
