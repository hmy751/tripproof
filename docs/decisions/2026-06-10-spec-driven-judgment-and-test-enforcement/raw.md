# Raw Notes - spec-driven 하네스: 문서 게이트에서 판단·테스트 강제로

이 파일은 `index.md` 결정의 배경 재료이며, 거쳐온 문제 감각과 교정을 보존하기 위한 자료다. 현재 실행 기준이나 작업 대기열이 아니다. 결론은 `index.md`에 있고, 여기 후보·표현을 현재 작업으로 쓰려면 `index.md`와 현재 코드 상태로 다시 판단한다.

## 왜 raw가 필요한가

이 결정은 "무엇을 바꿨나"보다 "어떤 직감에서 출발해 어디서 교정됐나"가 핵심이다. 출발 직감 몇 개는 그대로 맞지 않았고, 그 교정 과정을 남겨야 다음에 같은 논의가 반복될 때 출발점을 다시 헤매지 않는다.

## 출발 직감 (세 가지 개선 제안)

작업은 세 가지 직감에서 출발했다. 이 직감들은 독립 점검과 구분해 따로 정리했다.

1. **막는 개념(stub/slice)을 낮춘다.** stub/slice 어휘가 제품 개발 대신 목업·회피를 유도한다고 느꼈다. non-goals 수준으로만 남기고 개발을 방해하는 부분을 없애고 싶었다.
2. **시나리오 기반 spec을 완화한다.** 시나리오를 미리 다 잡으면 큰 틀에 갇힌다. 개발하다 정할 때 정하고, 시나리오는 참고용으로만 두고 개발에 무게를 둔다.
3. **하네스에 개발 감각을 보강한다.** LLM으로 풀 문제를 코드 내부 규칙으로 우회하는 패턴이 있었고, 이건 하네스의 개발 감각 부족 때문이라고 봤다.

## 교정 (직감이 어떻게 바뀌었나)

- **"기존이 다 틀린 게 아니다"(vintage).** 누적된 문서를 전부 불필요로 보는 전제를 버렸다. 최근에 박힌 guard 일부는 실제로 막 작동하기 시작했고, 초기 churn 흔적과 갈라야 한다.
- **증상이 아니라 판단을 개선한다.** 코드 내부 규칙 우회는 대표 증상일 뿐 타깃이 아니다. 그걸 잡는 탐지기를 만들면 걸리지 않는 다른 우회를 찾을 뿐이다. 개선 대상은 우회 자체가 아니라 그 선택을 만드는 결정 시점의 판단 감각이다.
- **세 번째 직감의 실체는 추상 규칙이 아니라 테스트.** 우회가 있다는 직감은 과거엔 맞았지만, 현재 라이브 코드는 이미 LLM을 제대로 부르고 코드 규칙은 틀린 답을 깎아내리는 역할만 한다. 그래서 "개발 감각"의 실체는 새 금지 규칙이 아니라 "자료가 바뀌면 답이 바뀌나 / 근거가 진짜 자료에 있나"를 테스트로 박는 것이었다.
- **강제력을 문서에서 테스트로.** 문서가 강제력을 떠안으니 게이트가 됐다. 테스트가 강제하면 문서는 가벼워진다.
- **구현은 최후 보루, 하네스로 근본을 푼다.** 문제를 구현 단계의 우회로 때우지 않고, 하네스를 고쳐 근본을 푸는 방향을 우선한다.

## 출처와 확인된 사실 (당시 코드 기준)

- 라이브 답변 경로는 LLM이 evidence_state를 먼저 정하고, 코드 내부 규칙(키워드·숫자 window)은 SUPPORTED 분기 안에서 검증 실패 시 MISSING으로 강등만 한다. MISSING→SUPPORTED 승격 경로는 없다. 따라서 "코드 규칙이 LLM을 우회한다"는 라이브 기준으로는 과장이었다. 단 과거(local 후보 생성 단계)에는 LLM 호출 자체를 생략한 drift가 실재했다 — "예전엔 우회 비슷했음"과 "지금 라이브는 우회 아님"을 섞지 않는다.
- 경계 관찰 implementation-note가 남은 뒤 며칠 안에 실제 코드 경로(local 후보 생성 → 실제 provider 호출)가 그 drift를 닫은 인과가 확인됐다. 교정 루프는 고장난 게 아니라 막 무는 중이다.
- 자동 강제장치(CI workflow, hook, pre-commit)는 0이고 `npm test` 수동 실행만 있다. 그래서 prose가 유일한 강제 수단이었다.
- 테스트가 생긴 뒤 문서를 다시 점검하니, 코드/테스트로 강제되는 두 줄도 단일 답변 경로만 덮어 자를 수 없었고 나머지는 판단 원칙이었다. 줄일 분량이 사실상 없다는 것은 분량이 문제가 아니었음을 뜻한다.

## 기각·보류된 후보 (현재 후보 아님)

- prose 전량 삭제 — 기각.
- 증상 탐지 lint/grep/CI 룰 — 기각.
- 행위자 감시 spy 테스트 — 보류/폐기(라이브에서 trivially green).
- 메타모픽의 전역 규칙 승격 — 보류.
- 시나리오 기반 spec을 더 푸는 별도 수술 — 이번 결정 밖. 발동 기준 축소까지만 하고, 시나리오 운영 자체는 기존 light loop 기준을 유지했다.

## 남은 불확실성

- 행위자 질문이 결정 시점 attention을 실제로 바꾸는지는 다음 새 도메인 구현에서 관찰해야 확인된다. 아직 product로 검증되지 않았다.
- 사례 앵커가 calibration으로 남을지, 선행 필독 게이트로 굳을지는 관찰이 필요하다.
- `docs/specs/README.md`의 동일 금지문과 SKILL의 질문형 사이 형식 불일치는 남겨뒀다.

## 재진입 메모

- 다음 큰 작업(특히 코드 내부 규칙이 아직 살아있는 채팅 답변 경로)을 시작할 때, 코드를 짜기 전에 행위자/변형 질문에 한 줄로 답하고, 같은 화면에 다른 자료를 넣어 답이 실제로 바뀌는지 한 번 관찰한다.
- 효과가 약하면 추가하지 않는다. 문제가 생기면 보완문 추가보다 직전 기준 복구를 먼저 본다.
