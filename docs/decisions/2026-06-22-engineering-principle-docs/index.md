# 2026-06-22 - 엔지니어링 판단 기준 문서(docs/engineering) 신설

## 폴더 구성

- `index.md`: 결론과 현재 판단.
- `raw.md`: 이 결정에 이른 출발 문제 감각, 거쳐온 교정, 논의 배경.

`raw.md`는 현재 실행 기준이 아니다. 다음 세션은 `index.md`를 먼저 읽고, 왜 이런 결정을 했는지 필요할 때만 `raw.md`를 본다.

## 맥락

이 repo는 그동안 하네스 문서의 역할을 낮추고(`2026-06-03` 하네스 문서 역할 재정렬), spec-driven 루프를 가볍게 했으며(`2026-06-05` 슬라이스 빌드 모델), 강제력을 문서에서 제품행동 테스트로 옮겼다(`2026-06-10` 문서 게이트에서 판단·테스트 강제로). 이 세 결정은 "문서가 작업을 어떻게 잡아먹지 않게 할까"와 "제품 동작을 어떻게 강제할까"를 다뤘다.

그런데 그 사이에 다른 종류의 마찰이 남아 있었다. 코드를 짜는 판단 자체 — 이름, 추상화를 만들 시점, 관심사 경계, 실패를 어디서 멈추고 어디서 낮출지 — 가 안정적으로 제어되지 않는다는 감각이다. 이 감각은 리팩토링 이력과 spec 문서를 보며 쌓였다. AI는 이번 작업을 가장 빨리 통과시키는 코드 쪽으로 기울고, 실제 행위자를 가리는 이름을 두며, 아직 쓰이지 않을 추상을 미리 만들고, prose 금지문은 코드를 짜는 그 순간의 판단을 바꾸지 못한다(`2026-06-10` 근거, `2026-06-09` 제품 흐름 drift, `2026-06-05` slice drift에서 같은 결로 관찰됨).

이 판단 기준은 이미 repo 안에 실천과 결정 기록으로 흩어져 있다. 그러나 코드를 짜기 직전에 항상 참고할 수 있는 단일 위치에는 모여 있지 않다. 작은 product 노트(`docs/development-notes.md`)가 일부를 담지만, 같은 내용이 `docs/product-model.md`·`docs/prd.md`·`eval/README.md` 등 여러 곳과 겹쳐 있고 개발 판단 기준으로서는 얇다.

## 근거

### 판단 기준을 한곳에 모으면 결정 시점에 참고할 단일 기준이 생긴다

- 이름·추상화 시점·관심사 경계·실패 정책 같은 판단은 테스트로 못 박기 어렵다. 흩어진 채로 두면 매 작업마다 같은 미끄러짐을 개별적으로 다시 발견한다(`2026-06-09`·`2026-06-10` 노트가 같은 종류의 drift를 매번 따로 기록한 것이 그 증거다).
- 항상 참고 가능한 위치에 모으면, 결정 직전 판단을 돕는 reference가 된다.

### 일반론이 아니라 이 repo의 실제 사례로 근거를 둔다

- 외부 원칙을 그대로 베낀 문장은 코드를 짜는 순간의 판단을 바꾸지 못한다(`2026-06-10`). 같은 실패를 피하려면 원칙마다 이 repo의 실제 코드·커밋·문서 사례를 근거로 붙인다.
- 이는 `2026-06-10`이 택한 calibration sample 방식(추상 규칙보다 같은 미끄러짐의 구체 사례를 참고)과 같은 방향이다.

### 이 문서는 강제 게이트가 아니라 판단 참고다

- `2026-06-03`은 하네스가 승인권이나 선행 관문이 아니라고 정했고, `2026-06-10`은 prose가 유일 강제장치가 되면 통과 게이트로 굳는다고 봤다. 새 문서가 또 다른 prose 게이트가 되면 그 실패를 반복한다.
- 그래서 이 문서의 성격은 reference다. 제품 행동의 강제는 계속 테스트가 맡고, 이 문서는 테스트가 못 잡는 판단(이름·추상화 시점·관심사 경계·실패 정책)을 다룬다.

### 외부 레퍼런스는 정의째 도입하지 않고 어휘로 빌린다

- `2026-06-05`가 외부 spec-driven 용어를 한 번 확인하고 repo 규모에 맞게 정렬한 것과 같은 방식으로, 외부 엔지니어링 원칙도 이름과 calibration 감각만 빌리고 무거운 정의 체계는 들이지 않는다.

## 결정

코드 개발 판단 기준을 `docs/engineering/`에 모은다.

### 문서 구성

- `docs/engineering/principle.md`: 개발 판단의 핵심 기준. 다음 사람이 읽고 바꾼다는 관점, 변경을 작게·구조와 동작을 분리, 관심사 경계, 추상화를 만들 시점, 실패와 불확실성 설계 등.
- `docs/engineering/architecture.md`: 모듈 지도와 product / 관측 / eval 경계.
- `docs/engineering/testing.md`: 검증 종류, 테스트 크기, LLM·근거 기반 제품의 특수성.
- `docs/engineering/ai-coding.md`: AI가 코드를 짤 때 반복되는 미끄러짐과 그 경계.
- `docs/engineering/README.md`: 폴더 입구와 기록 기준.

### 성격과 범위

- 이 문서들은 항상 참고 가능한 판단 기준이다. 선행 게이트가 아니다. 제품 행동의 강제는 테스트가 맡는다.
- 각 원칙은 이 repo의 실제 사례를 근거로 둔다. 일반론 나열에 그치지 않는다.
- 외부 레퍼런스는 어휘·calibration으로 빌리되 repo 맥락에 맞게 정렬한다.
- 읽고 바꾸기 쉬움(다음 사람이 따라올 수 있는가)은 별도 문서가 아니라 `principle.md`의 목적으로 둔다.

### development-notes 해체

- `docs/development-notes.md`는 해체하고 내용을 분배한 뒤 제거한다. "기본 방향"(근거 부족·충돌이면 그 상태를 드러냄)은 `principle.md`의 실패·불확실성 설계로, product/eval 경계는 `architecture.md`로, 기록 기준은 `docs/engineering/README.md`로 옮긴다.

### 항상 켜지는 문서

- `CLAUDE.md`에 `docs/engineering/`를 가리키는 얇은 포인터 한 줄을 둔다. 운영 규칙 본문은 `CLAUDE.md`에 추가하지 않는다(`2026-06-03` 경계 유지).

## 기각 또는 보류

- **원칙 문서를 통과 게이트로 두기 — 기각.** prose가 유일 강제장치가 되면 게이트로 굳는 실패(`2026-06-10`)를 반복한다. reference로만 둔다.
- **읽기 쉬움(DX)을 별도 문서로 두기 — 기각.** 별도 항목군이 아니라 `principle.md`가 봉사하는 목적이다.
- **외부 레퍼런스를 정의 체계째 도입 — 기각.** 현재 규모에 과하다. 이름과 calibration만 빌린다(`2026-06-05`와 같은 방향).
- **`development-notes.md` 보존 — 기각.** 같은 내용이 여러 문서와 겹쳐 얇은 고아 문서로 남았다. 분배 후 제거한다.
- **조사에서 제안된 추가 원칙을 전부 1급으로 올리기 — 기각/추림.** 제안 다수가 기존 원칙과, 또는 서로 간에 같은 내용을 다른 이름으로 중복했다. 도메인 특수 빈칸만 추가한다(추출 손실과 한국어 단위 grounding, 관심사가 잘못된 축에 붙는 오염, 틀린 대상을 지우지 않고 주의문만 덧대는 보완).
- **CI/hook 같은 자동 게이트 — 보류.** 이번 결정 밖이며 도입 시점은 별도로 본다(`2026-06-10`과 같은 경계).

## 검증

- 동기는 repo 자체 기록으로 뒷받침된다: `2026-06-03`(문서가 현재 명령처럼 읽힘, 덧대기식 보완), `2026-06-05`(얇게가 빠른 결과로 미끄러짐), `2026-06-09`(얇게를 vertical slice 아닌 가장 빨리 보이는 결과로 해석), `2026-06-10`(prose는 결정 순간의 attention을 못 바꾸고, 모델은 가장 빨리 통과시키는 코드로 기운다).
- 각 원칙이 이 repo의 실제 코드·커밋·문서에 근거를 갖는지 확인했다. 예: 이름과 계약이 어긋난 부분 정정(`fact_proposer`→`answer_composer`, `facts.py`→`evidence.py`), 리팩터 커밋의 "동작 무변경/동작 변경" 표기, `EvidenceState`가 불확실성을 정상 상태로 모델링, snippet이 원문의 정확한 부분일 때만 근거가 만들어지는 grounding 계약, 관측의 "내부 record 먼저, exporter 나중" 경계.
- 정리한 예시의 인용은 코드·커밋·문서에서 직접 확인했고 사실관계 오류는 없었다.
- 원칙 문서 본문 작성은 이 결정의 후속 작업이다. 이 결정은 위치·성격·범위를 정한다.

## 이번 결정 밖

- `docs/engineering/` 각 문서의 세부 작성과 `development-notes.md` 실제 제거.
- `CLAUDE.md` 포인터 문구의 최종 표현.
- CI/hook 같은 자동 게이트 도입.
- 개별 코드 후속(아직 도달하지 않는 상태값 처리 등)은 다루지 않는다.

## 본문 작성 기준 (후속)

"각 원칙은 repo 실제 사례를 근거로 둔다"는 의도만으로는 본문이 추상적 원칙 나열로 흐르기 쉽다. 본문을 쓰고 고칠 때 따르는 기준을 후속으로 남긴다.

- 실패 모드는 원칙이 고상해서가 아니라, 맨 라벨과 당위 선언에 결정 시점 트리거와 코드에서 알아챌 신호가 없는 것이다.
- 각 원칙은 이 repo에서 실제로 틀리는 분기로 적는다. "X 하려는 순간이면 Y"라는 트리거, 격언은 "원인 → 나쁜 결과", 과밀한 줄은 분해, 충돌하는 짝(빨리 vs 읽힘, 분리 vs 단순)에는 결선 규칙을 둔다. 외부 라벨(YAGNI·DIP 등)은 회상 앵커로 괄호에 두고 본문이 자립하게 한다.
- 발화 모범의 수준은 한국어 단위 grounding이나 "얇게 = 가장 빨리 보이는 결과"처럼, 추상 라벨이 아니라 결정 시점에 작동하는 구체 신호다. 본문의 떠 있는 원칙을 이 수준에 맞춘다.
- 과교정은 금한다. 모든 원칙에 좋음·나쁨·냄새 세트를 강제하지 않는다. 이미 발화하는 줄은 그대로 두고 죽은 줄에만 최소로 보강한다. 자기 점검 표현은 두되 통과 게이트로 만들지 않는다.
