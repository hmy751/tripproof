# Raw Notes - 엔지니어링 판단 기준 문서(docs/engineering) 신설

이 파일은 `index.md` 결정의 배경 재료이며, 거쳐온 문제 감각과 교정을 보존하기 위한 자료다. 현재 실행 기준이나 작업 대기열이 아니다. 결론은 `index.md`에 있고, 여기 후보·표현을 현재 작업으로 쓰려면 `index.md`와 현재 코드 상태로 다시 판단한다.

## 왜 raw가 필요한가

이 결정의 핵심은 "어떤 문서를 만든다"보다 "왜 또 문서를 만드는데도 그것이 이전에 실패한 prose 게이트가 되지 않는가"다. 출발 동기와 그 사이의 교정을 남겨야, 다음에 같은 문서를 손볼 때 "원칙을 더 적자"는 출발점으로 돌아가 같은 실패를 반복하지 않는다.

## 대화·조사에서 드러난 문제 감각

### 출발 동기

- 개발 규칙·원칙을 어디에 두는가에서 시작했다. skill에 두는 것은 맞지 않았다. 항상 참고하는 판단 기준은 늘 읽히는 문서층에 두는 것이 자연스럽다.
- 더 근본적인 동기는, 리팩토링 이력과 spec 문서에서 AI의 코드 판단이 안정적으로 제어되지 않는 패턴이 반복된 것이다. 이름·추상화 시점·관심사 경계 같은 판단이 매번 흔들렸다.

### 거쳐온 교정

- **기존 구조는 존재한다는 사실만으로 따르지 않는다.** 어떤 문서나 기록이 이미 있다는 것이 그 구조가 잘 작동한다는 근거는 아니다. 다만 이는 전면 재작성을 뜻하지 않는다(아래 보완 원칙).
- **외부 엔지니어링 원칙은 정의 체계째 도입하지 않고, 이름과 calibration 감각만 골라 repo 규모에 맞게 정렬한다.**
- **읽고 바꾸기 쉬움(DX)은 별도 항목군이 아니라 원칙 문서가 봉사하는 목적이다.** AI가 결과를 빨리 만들지만 사람이 따라가기 어렵다는 우려는, 별도 문서가 아니라 코드를 짜고 이름을 붙이는 그 자리에서 녹아야 한다.
- **기존 판단 구조(원칙 목록의 골격)는 유지하고, repo 조사는 그것을 평가·제거하는 용도가 아니라 보완하는 용도로 쓴다.** 조사로 골격을 다시 평가하거나 외부 원칙을 제거하는 방향은 현재 규모에 맞지 않는다. 골격은 두고, 각 원칙에 이 repo의 실제 사례를 붙이고 빠진 것만 채운다.
- **추가 원칙 후보를 추렸다.** 추가 원칙 후보를 조사한 결과 상당수가 기존 원칙과, 또는 서로 간에 같은 내용을 다른 이름으로 중복했다. 중복은 합쳐 보조 예시로 낮추고, 도메인 특수 빈칸만 새 항목으로 올렸다.

## 출처와 확인된 사실

- 동기는 repo 자체 결정·노트로 뒷받침된다.
  - `2026-06-03` 하네스 문서 역할 재정렬: 문서가 현재 명령·작업 대기열·승인권처럼 읽히는 문제, 그리고 틀린 대상을 지우는 대신 주의문만 덧대는 "덧대기식 보완".
  - `2026-06-05` 슬라이스 빌드 모델: "얇게"가 가장 빨리 보이는 결과로 미끄러지는 drift.
  - `2026-06-09` 제품 흐름 drift: AI가 "얇게"를 vertical slice가 아니라 가장 빨리 보이는 결과로 해석할 수 있다는 관찰.
  - `2026-06-10` 문서 게이트에서 판단·테스트 강제로: prose는 코드를 짜는 그 순간의 attention을 바꾸지 못하고, 모델은 이번 작업을 가장 빨리 통과시키는 코드 쪽으로 기운다. 그래서 강제력을 제품행동 테스트로 옮기고 문서는 판단을 돕는 reference로 둔다.
- 정리한 원칙과 예시는 그것을 비판적으로 다시 보고, 인용이 실제 코드·커밋·문서와 맞는지 확인하는 과정을 거쳤다. 사실관계 오류는 발견되지 않았다.
- repo는 이미 자기 판단 기준을 실천하고 일부는 결정으로 명문화하고 있다(분리는 모양이 아니라 정당성으로 판단, 이름은 담는 계약을 말한다, 관측은 내부 record가 먼저 등 — `2026-06-19` 결정). 새 문서는 이 실천을 발명하는 것이 아니라 한곳에 모으고 근거를 붙인다.

## 기각·보류된 후보 (현재 후보 아님)

- 원칙 문서를 통과 게이트로 두기 — 기각.
- 읽기 쉬움(DX)을 `docs/engineering/` 아래 별도 문서로 두기 — 기각(원칙 문서의 목적으로 흡수).
- 외부 레퍼런스를 정의 체계째 도입 — 기각.
- 조사 결과로 기존 원칙 골격을 다시 심판·축소 — 기각(조사는 보완용).
- 제안된 추가 원칙을 전부 1급으로 올리기 — 기각(중복은 합치고 도메인 특수 빈칸만 추가).
- `development-notes.md` 보존 — 기각(분배 후 제거).
- CI/hook 자동 게이트 — 보류(이번 결정 밖).

## 남은 불확실성

- 원칙 문서가 실제로 결정 시점의 판단을 바꾸는지는 다음 작업에서 관찰해야 확인된다. `2026-06-10`이 같은 질문을 열어 두었다 — reference가 결정 직전 attention을 바꾸는지, 아니면 다시 읽히지 않는 prose가 되는지.
- 추가한 도메인 특수 빈칸(추출 손실·한국어 단위 grounding 등)을 어느 문서 한 곳에 둘지는 본문 작성에서 고정해야 또 흩어지지 않는다.

## 재진입 메모

- 다음 단계는 `docs/engineering/` 각 문서 본문 작성과 `development-notes.md` 제거다.
- 각 원칙에는 이 repo의 실제 사례를 한 개 이상 붙인다. 일반론만 적지 않는다.
- 문서를 손볼 때 "원칙을 한 줄 더 적자"가 첫 반응이 되면 멈춘다. 강제는 테스트가 맡고 이 문서는 reference라는 경계를 먼저 확인한다.
- 본문에는 논의 과정에서 쓰던 임시 표현을 들이지 않고 프로젝트 문서 어투로 쓴다.
