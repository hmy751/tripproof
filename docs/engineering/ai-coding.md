# AI 코딩 규율

AI가 코드를 만들 때의 기준이다. AI는 빠르게 생성하느라 기존 코드 관습을 무시하고 자기 방식으로 짜기 쉽다. 사람이 따라올 수 있게 짜는 것이 핵심이다.

## 기존 코드에 맞춰 짠다

- 새 코드를 쓰기 전에 주변 코드의 이름·관습·추상화 수준을 먼저 읽고 거기 맞춘다. 내 선호 스타일을 들이지 않는다.
- 같은 일을 이미 하는 곳이 있으면 그 방식을 따른다.

## 자주 빠지는 실패

- 요청한 범위를 넘겨 "나중을 위해" 더 짠다 (`principle.md`의 YAGNI).
- "얇게"를 vertical slice가 아니라 가장 빨리 보이는 결과로 해석한다. vertical slice는 한 자료에 대해 입력 → 검색 → 근거 → 응답이 끝까지 한 번 도는 것이다. route만 만들고 use_case를 stub으로 두거나 UI만 먼저 올리는 건 빨리 보이는 결과지 slice가 아니다.
- 이름·라벨이 실제 행위자를 가린다. `validate_and_save()`가 변환까지 하거나 `helper`가 핵심 로직을 들고 있으면 라벨이 행위를 가린 것이다.
- 고치기 전에 주변을 충분히 읽지 않는다.

## 계획을 구현처럼 다루지 않는다

- 아직 없는 흐름이나 결과를 있는 것처럼 쓰지 않는다.
- 과거 분석·로드맵·노트를 그대로 다음 명령으로 승격하지 않는다.

## 틀린 것은 덧대지 말고 정리한다

- 틀린 대상은 지우거나 고치거나 격리한다. "이건 기준 아님" 같은 주의문만 덧대면 다음 사람이 그것을 다시 근거로 읽는다.

## 코드와 런타임 답을 구분한다

- "AI가 짠 코드"와 "AI가 런타임에 만든 답"은 다른 문제다. AI가 짠 코드는 사람이 짠 코드와 같은 기준으로 읽고 고친다. LLM이 런타임에 만든 답은 못 믿을 입력으로 보고 제품 코드가 근거로 검증한다 — 이 구분을 흐리지 않는다.
