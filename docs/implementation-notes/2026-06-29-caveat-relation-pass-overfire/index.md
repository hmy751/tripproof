# 2026-06-29 - caveat relation pass가 gemma3:4b에서 과잉강등

## 폴더 구성

- `index.md`: 독립적으로 읽히는 관찰과 다음에 볼 경계.
- `raw.md`: eval run 16~19의 출처와 질문별 측정값(과잉 부착된 caveat 발췌 포함). run artifact가 gitignore라 핵심 수치를 여기 보존한다.

## 왜 남기나

`06`의 방향(값을 좌우하는 caveat를 의미 층이 찾아 코드가 읽어 강등)을 코드로 구현해 본 결과, **gemma3:4b가 무관한 조건을 "이 답을 제한한다"고 과잉으로 갖다 붙여** 깨끗한 답까지 `needs_review`로 내렸다. 프롬프트·순서 트릭(per-unit, 순서 불변)으로도 못 잡았다. 다음 구현이 같은 천장에 다시 부딪히지 않도록, 무엇을 시도했고 왜 안 됐는지를 남긴다.

## 거쳐온 버전 (순서)

1. **조건 안 봄 (mechanical only, run 15)** — 답+근거만. 깨끗하나 P1-01 위험은 LLM 출력 운에 맡김.
2. **답과 함께 caveat 묻기 (run 16, commit `5d1880f`)** — 한 호출이 답+조건. 모델이 조건 칸을 안 채워 P1-01 놓침.
3. **caveat 따로 묻기 = 분리 호출 (run 17, commit `574dee4`)** — 답 호출과 별개로 "조건 있냐"만 호출. 과잉(엉뚱한 조건 부착).
4. **per-unit(B) / 순서 불변(A) (run 18·19, commit `118a916`)** — 분리 호출을 다듬음. 여전히 과잉. 되돌림(`8040665`).

## 관찰 (상세 수치는 `raw.md`)

- **per-unit(B)은 P1-01 정밀도를 살렸다.** run 17은 P1-01에 노쇼 조건이 잘못 붙었는데, source unit마다 따로 판정하니(B) **정확한 조건**("체크인 시 숙소 측의 상황에 따라 반영 여부가 결정됩니다")을 붙여 `needs_review`로 내렸다.
- **그래도 깨끗한 답의 과잉은 남았다.** B·A 모두 날짜에 취소정책, 객실/인원·체크인에 신분증 정책을 "제한 조건"으로 붙였다. gemma3:4b가 그렇게 *판단*한다.
- **순서 불변(A)은 과잉을 못 걸렀다.** 막으려던 무관 부착이 두 순서 모두에서 일관되게 나왔다 — 우연한 noise가 아니라 *일관된 오판*이라 교차검증을 그대로 통과했다.
- **단일 run 변동이 비교를 압도한다.** 같은 seed인데 answer composer 자체가 run마다 흔들려(상태·item 개수 변동), A↔B 우열을 단일 run으로 못 가린다(03 재현성 원칙).

핵심: 막힌 지점은 프롬프트/순서 설계가 아니라 **gemma3:4b가 "이 조건이 *이 답을* 제한하나"를 정밀히 못 가르는 것** — 모델 정밀도 천장이다. 과잉강등은 *위험한 쪽(confident-wrong)이 아니라 안전한 쪽*이라 치명적이진 않지만, 깨끗한 lookup을 불필요하게 내려 제품 사용성을 깎는다.

## 결정 (이번 관찰로 정한 것)

- per-unit·순서 불변 실험(`118a916`)은 **되돌렸다(`8040665`)**. 측정으로 효과를 못 봤기 때문.
- **분리 호출 baseline(`574dee4`)은 유지**한다 — `06`의 구조(답 호출과 분리된 caveat 검출)는 살려두고, 검출 품질만 모델/접근이 받쳐줄 때 다시 끌어올린다.
- 이 baseline도 과잉강등이 있다는 것을 알려진 한계로 둔다(안전한 쪽 실패라 일단 ON 유지). 끄거나 mechanical only로 더 후퇴할지는 별도 판단.

## 다시 볼 경계

caveat/조건 검출을 구현할 때 멈춰서 확인한다.

- 작은 모델(gemma3:4b)에 "이 조건이 이 답을 제한하나"를 맡기면 무관한 조건을 과잉 부착한다. 프롬프트를 per-unit으로 좁히거나 순서 불변으로 교차검증해도 *일관된 오판*은 안 걸린다. 이건 모델 정밀도 문제지 프롬프트 문제가 아니다.
- 다시 도전한다면: per-unit(B)이 P1-01 정밀도에선 더 나았으니 거기서 출발하고(`118a916` 참고), 검출만 더 강한 모델로 돌리거나(생성/검증 분리 유지), repeat run으로 noise부터 걷어낸 뒤 본다.
- 과잉강등이 "안전한 쪽"이라고 방치하면, run 14식 "전부 needs_review"로 제품이 못 쓰게 되는 길과 가깝다. 깨끗한 값 lookup이 얼마나 내려가는지를 항상 같이 본다.
