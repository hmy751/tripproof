# TripProof PRD

상태: 초안. 이 문서는 제품 요구사항을 정리하기 위한 기준 문서이며, 구현 완료나 평가 결과를 주장하지 않는다.

## Document Scope

이 문서는 TripProof의 첫 제품 요구사항을 사용자 문제, 사용자 흐름, 기능 요구, AI 동작 기준으로 정리한다. 기존 논의에서 나온 판단과 아이디어는 문서 안에서 작성 과정으로 설명하지 않고, 아래 기준에 맞춰 제품 요구사항으로 흡수한다.

현재 repo 구조는 `src/client/`, `src/server/`, `src/server/ai/`, `src/shared/`를 기준으로 한다. 이 PRD는 별도 `src/product/` wrapper나 eval artifact를 전제하지 않는다.

| 구분 | 이 PRD에서의 의미 | 적용 방식 |
| --- | --- | --- |
| 제품 방향 | 이번 버전에서 지켜야 할 판단 기준 | 사용자 흐름, 근거 중심 확인, 사용자가 이해할 수 있는 상태 언어를 우선한다. |

## 1. Product Overview

TripProof는 여행자가 받은 예약서, 바우처, 안내 메시지, 약관, 영수증 같은 자료에서 행동 전에 필요한 정보를 원문 근거와 함께 확인하도록 돕는 여행 자료 검증 제품이다.

대상 사용자는 여행 전후로 여러 자료를 받았고, 체크인, 투어 출발, 렌터카 반납, 결제 확인처럼 실수 비용이 큰 순간에 특정 정보를 빠르게 확인해야 하는 여행자다.

TripProof가 해결하려는 핵심 문제는 "자료를 찾기 어렵다"가 아니라 "필요한 정보가 어떤 자료에 근거하는지, 그 정보가 충분히 확정 가능한지, 자료끼리 충돌하지 않는지 사용자가 판단하기 어렵다"는 점이다.

제품의 핵심 가치는 다음 네 가지다.

- 필요한 순간에 필요한 fact를 먼저 보여준다.
- fact마다 원문 근거를 연결한다.
- 근거 부족, 확인 필요, 자료 충돌, 직접 확인 같은 상태를 숨기지 않는다.
- 사용자가 확인한 정보만 저장된 여행 정보로 승격한다.

## 2. Problem Statement

여행자는 여행 준비와 이동 과정에서 여러 형태의 자료를 받는다.

- 호텔 예약 확인서
- 호스트 체크인 안내
- 투어 바우처
- 교통 티켓
- 렌터카 약관
- 보험 약관
- 결제 영수증
- 앱 화면이나 메시지 캡처

기존 방식의 한계는 다음과 같다.

- PDF 검색이나 메신저 검색은 사용자가 어떤 키워드로 찾아야 하는지 알고 있어야 한다.
- 일반 챗봇 답변은 답의 근거, 근거 부족, 자료 충돌을 충분히 드러내지 않을 수 있다.
- 문서 보관함은 자료를 모아두지만, 지금 행동에 필요한 정보와 확인 상태를 구분해주지 않는다.
- 사용자가 승인하지 않은 AI 추출값이 저장 정보처럼 보이면 잘못된 행동으로 이어질 수 있다.

TripProof가 다루는 문제 범위는 여행 자료에서 행동 가능한 fact를 추출하고, 근거와 상태를 함께 보여주며, 사용자의 확인을 거쳐 저장된 정보로 만드는 흐름이다.

이번 PRD에서 다루지 않는 범위는 다음과 같다.

- 여행 일정 자동 생성
- 항공권, 숙소, 투어 예약 또는 결제
- 모든 여행 자료를 저장하는 범용 문서 금고
- 일반 목적 PDF 챗봇
- 제품 품질 점검용 대시보드
- 다중 사용자, 권한 관리, 배포 운영

## 3. Goals / Non-goals

### Goals

- 사용자가 여행 자료 묶음을 입력하고 특정 여행 순간을 선택할 수 있다.
- TripProof가 그 순간에 필요한 핵심 fact 후보를 구조화해서 제시한다.
- 각 fact 후보는 원문 근거와 연결된다.
- 근거가 부족한 항목은 답을 만들어내지 않고 `근거 부족` 또는 `직접 확인` 상태로 표시한다.
- 서로 다른 자료가 같은 항목을 다르게 말하면 하나로 합치지 않고 `자료 충돌` 상태로 표시한다.
- 예약번호, 출입 코드, 연락처, 결제 정보처럼 민감하거나 위험한 값은 자동 저장하지 않는다.
- 사용자가 fact 후보를 저장, 수정, 무시할 수 있다.
- 저장된 fact만 이후 상황 카드나 확인 결과에 반영된다.
- 제품 품질은 답이 자료와 맞는지, 근거가 정확한지, 근거 부족에서 멈추는지, 자료 충돌을 드러내는지의 관점에서 점검 가능해야 한다.

### Non-goals

- 첫 버전에서 모든 여행 순간을 지원하지 않는다.
- 첫 버전에서 실제 파일 인식이나 이미지 영역 근거 표시 전체를 완성하지 않는다.
- AI가 추출한 정보를 사용자 확인 없이 최종 여행 정보로 저장하지 않는다.
- 품질 점검 지표, 실행 기록, 내부 진단 정보를 사용자 화면의 중심 언어로 쓰지 않는다.
- 공개용 품질 수치나 개선 전후 상태를 실제 실행 전 문서에 쓰지 않는다.

### MVP 기준

MVP는 "여행 자료 검증 제품"의 전체 카탈로그가 아니라, 한 사용자 순간을 끝까지 통과시키는 얇은 흐름이다.

MVP 시나리오는 숙소 체크인 확인이다. P0에서는 예약 확인서와 호스트 안내 같은 자료를 넣고 체크인 시작 시간과 늦은 도착 조건을 확인한다. 예약번호 필요 여부와 출입 방법 전체 추출은 후속 확장으로 둔다.

이 시나리오는 최종 제품 전체를 대표하지는 않지만, TripProof의 핵심 요구사항인 구조화된 fact, 원문 근거, 근거 부족, 자료 충돌, 사용자 검수, 저장된 정보 반영을 작게 검증하기에 적합하다.

## 4. Target User & Use Cases

### Target User

주요 사용자는 직접 여행 자료를 관리하고, 여행 중 중요한 조건을 스스로 확인해야 하는 개인 여행자다.

이 사용자는 다음 상황에 자주 놓인다.

- 이동 중이라 긴 문서를 다시 읽기 어렵다.
- 여러 앱, 이메일, PDF, 메시지에 정보가 흩어져 있다.
- 잘못 알면 체크인 실패, 투어 지각, 추가 결제, 보증금 문제 같은 비용이 생긴다.
- AI가 답을 줘도 근거가 없으면 믿기 어렵다.

### Use Cases

| Use case | 사용자 상황 | 확인할 정보 |
| --- | --- | --- |
| 숙소 체크인 | 숙소 근처에 도착했거나 이동 중이다. | 체크인 시간, 늦은 도착 조건, 예약번호 필요 여부, 출입 방법 |
| 투어 출발 | 투어 시작 직전 바우처와 안내 메시지를 확인한다. | 집합 시간, 집합 장소, 모바일 바우처 가능 여부, 지각 규정 |
| 렌터카 반납 | 차량 반납 전 약관과 예약 조건을 확인한다. | 연료 정책, 보증금 조건, 반납 위치, 반납 시간 |
| 여행 후 결제 확인 | 여행 후 영수증과 예약 조건을 대조한다. | 추가 결제 이유, 도시세, 보증금 환불 조건, 취소/환불 규정 |

MVP는 위 use case 중 하나를 먼저 깊게 닫는다. 나머지는 같은 구조를 확장할 후보로 둔다.

## 5. Core User Flow

### MVP Flow

1. 사용자가 여행 순간을 선택한다.
   - 예: `숙소 체크인`
2. 사용자가 관련 자료를 추가한다.
   - 예: `호텔 예약 확인서`, `호스트 체크인 안내`, `앱 메시지 캡처`
3. 사용자가 확인하고 싶은 질문이나 체크리스트를 선택한다.
   - 예: `체크인은 몇 시부터 가능한가?`, `늦게 도착하면 어떻게 해야 하는가?`
4. TripProof가 자료에서 관련 fact 후보를 추출한다.
5. 각 fact 후보에 원문 근거를 연결한다.
6. TripProof가 fact별 상태를 표시한다.
   - `확인됨`
   - `확인 필요`
   - `근거 부족`
   - `자료 충돌`
   - `직접 확인`
7. 사용자가 후보 fact를 검토한다.
   - 저장
   - 수정 후 저장
   - 무시
   - 직접 확인 표시
8. 저장된 fact만 상황 카드에 반영된다.
9. 사용자는 상황 카드에서 행동에 필요한 최종 확인 결과를 본다.

### Expected Result

사용자는 긴 자료 전체를 다시 읽지 않고도, 지금 행동에 필요한 정보와 그 근거, 그리고 확정할 수 없는 항목을 한 화면에서 확인할 수 있어야 한다.

## 6. Feature Requirements

### MVP Features

| Feature | 목적 | 기대 동작 |
| --- | --- | --- |
| 여행 순간 선택 | 자료 전체가 아니라 사용자의 현재 행동 맥락을 기준으로 확인 범위를 좁힌다. | 사용자는 `숙소 체크인` 같은 상황을 선택하고, 시스템은 그 상황에 맞는 확인 항목을 우선한다. |
| 자료 묶음 입력 | 흩어진 자료를 한 확인 흐름 안에 넣는다. | 사용자는 여러 자료를 이름과 함께 추가할 수 있다. 각 자료는 이후 근거 표시에서 자료명으로 식별된다. |
| 핵심 fact 추출 | 질문에 필요한 정보를 자유 답변이 아니라 검토 가능한 후보로 만든다. | 시스템은 체크인 시간, 늦은 도착 조건, 출입 방법 같은 항목을 fact 후보로 제시한다. |
| 원문 근거 연결 | 사용자가 답이 나온 자료와 원문 위치를 확인할 수 있게 한다. | 각 fact에는 자료명과 원문 발췌 또는 위치 단서가 붙는다. |
| 상태 분류 | 자동 확정해도 되는 항목과 사용자가 다시 봐야 하는 항목을 나눈다. | fact는 `확인됨`, `확인 필요`, `근거 부족`, `자료 충돌`, `직접 확인` 중 하나의 상태를 가진다. |
| 충돌 표시 | 서로 다른 자료의 값을 조용히 합치지 않는다. | 같은 항목에 다른 값이 있으면 양쪽 근거를 보여주고 `자료 충돌`로 표시한다. |
| 민감 정보 보호 | 위험한 값이 자동 저장되거나 외부에 노출되는 진단 기록에 새는 것을 막는다. | 예약번호, 출입 코드, 연락처, 결제 정보는 자동 저장하지 않고 확인 대상으로 둔다. |
| 사용자 검수 | AI 추출값을 사용자가 승인 가능한 단위로 만든다. | 사용자는 fact 후보를 저장, 수정 후 저장, 무시할 수 있다. |
| 상황 카드 | 확인된 정보만 행동 가능한 결과로 모은다. | 저장된 fact만 상황 카드에 표시되고, 저장되지 않은 후보는 최종 정보처럼 보이지 않는다. |

## 7. Acceptance Criteria

### Material Input

- 사용자는 최소 두 개 이상의 자료를 이름과 함께 추가할 수 있다.
- 각 자료는 이후 fact 근거에서 자료명으로 식별 가능해야 한다.
- 자료명이 없는 자료는 사용자가 구분 가능한 기본 이름을 받아야 한다.

### Fact Extraction

- 시스템은 선택된 여행 순간에 필요한 fact 후보를 제시해야 한다.
- fact 후보는 이름, 값, 상태, 근거를 가진다.
- 자료에 없는 값은 일반 지식으로 채우지 않는다.
- 자료에 있는 표현이 애매하면 `확인 필요` 또는 `직접 확인` 상태로 둔다.

### Evidence

- `확인됨` 상태의 fact는 최소 하나 이상의 원문 근거를 가져야 한다.
- 근거는 사용자가 원자료에서 다시 찾을 수 있을 만큼 구체적이어야 한다.
- 답변에 쓰인 값과 원문 발췌가 서로 맞지 않으면 완료로 보지 않는다.

### Insufficient Evidence

- 사용자가 묻는 정보가 자료에 없으면 시스템은 값을 만들어내지 않는다.
- 이 경우 결과는 `근거 부족`으로 표시되고, 왜 답할 수 없는지 짧게 설명한다.
- `근거 부족` 항목은 사용자가 직접 입력하거나 확인하기 전까지 상황 카드에 저장 정보로 표시되지 않는다.

### Conflict

- 같은 항목에 대해 서로 다른 자료가 다른 값을 말하면 시스템은 하나를 선택하지 않는다.
- 충돌한 값과 각 값의 근거가 함께 표시되어야 한다.
- 사용자가 직접 선택하거나 수정하기 전까지 충돌 항목은 최종 저장 정보로 승격되지 않는다.

### Review / Save

- 사용자는 fact 후보를 저장, 수정 후 저장, 무시할 수 있다.
- 저장된 fact만 상황 카드에 표시된다.
- 무시한 fact는 최종 확인 결과처럼 보이지 않는다.
- 수정 후 저장된 fact는 원래 AI 후보와 사용자가 수정한 값이 구분 가능해야 한다.

### Sensitive Information

- 예약번호, 출입 코드, 연락처, 결제 정보 같은 민감 값은 자동 저장하지 않는다.
- 민감 값이 원문 발췌에 포함될 경우 표시 수위나 masking 여부를 검토해야 한다.
- 민감 값은 공개 가능한 예시나 품질 리포트에 그대로 포함되지 않아야 한다.

### AI Failure Cases

- 근거 없는 답변 생성은 실패다.
- 근거 위치가 실제 답의 근거를 가리키지 않으면 실패다.
- 자료 충돌을 단일 답으로 뭉개면 실패다.
- 민감 정보를 자동 확정하거나 저장하면 실패다.
- 내부 평가 용어만 노출되어 사용자가 다음 행동을 이해하지 못하면 실패다.

## 8. AI Behavior Requirements

### AI Inputs

AI는 다음 입력을 기준으로 동작해야 한다.

- 사용자가 선택한 여행 순간
- 사용자의 질문 또는 확인 항목
- 사용자가 제공한 자료 묶음
- 각 자료의 이름
- 이미 저장된 fact가 있다면 그 목록
- 사용자가 직접 수정하거나 확인한 값

### AI Outputs

AI는 자유 답변보다 검토 가능한 구조를 우선 출력해야 한다.

각 출력 항목은 다음을 포함한다.

- fact 이름
- fact 값 또는 `null`
- evidence state 또는 사용자-facing status
- 근거 목록
- status reason
- sensitive 여부
- conflict 관계가 있다면 충돌한 후보 목록

### Uncertainty Handling

AI는 불확실한 정보를 확정값처럼 말하면 안 된다.

- 자료에 근거가 없으면 `근거 부족`으로 둔다.
- 자료의 표현이 조건부이면 조건을 함께 표시한다.
- 날짜, 시간, timezone, 금액, 환불 조건이 애매하면 `확인 필요`로 둔다.
- 여러 자료가 충돌하면 `자료 충돌`로 둔다.
- 사용자가 최종 판단해야 하는 경우 `직접 확인`으로 둔다.

### User Confirmation Required

다음 상황에서는 사용자 확인이 필요하다.

- 예약번호, 출입 코드, 연락처, 결제 정보가 포함된 경우
- 자료끼리 같은 항목을 다르게 말하는 경우
- 근거는 있지만 조건이 복잡하거나 예외가 있는 경우
- AI가 값은 추출했지만 근거 위치가 약한 경우
- 사용자가 직접 수정한 값이 기존 근거와 다를 경우

### AI Must Not

- 자료 밖 일반 지식으로 여행 조건을 채우지 않는다.
- 두 자료의 충돌을 임의로 해결하지 않는다.
- 민감 정보를 자동 저장하지 않는다.
- 사용자가 저장하지 않은 fact를 상황 카드에 최종 정보처럼 표시하지 않는다.
- 내부 진단 용어나 구현 타입명을 사용자 설명의 중심에 두지 않는다.
- 구현되지 않은 자료 인식, 충돌 분석, 민감 정보 정책, 품질 점검 결과를 완료된 기능처럼 표현하지 않는다.

## 9. Data / Domain Model

이 섹션은 구현 언어의 확정 타입이 아니라 제품 요구를 설명하기 위한 도메인 모델이다.

### TravelMoment

사용자가 확인하려는 여행 상황이다.

예:

- accommodation_checkin
- tour_departure
- rental_return
- post_trip_payment_review

### TravelArtifact

사용자가 제공한 원자료의 메타 정보다. 현재 공유 타입 이름은 `TravelArtifact`다.

현재 또는 가까운 후보 필드:

- id
- name
- fileName
- kind
- content 또는 text reference
- createdAt 또는 receivedAt
- redactionStatus

P0 synthetic text는 server input이나 fixture case에서 `artifact.id`와 연결한다. 실제 파일 content와 OCR 결과를 공유 타입에 직접 넣을지는 파일 ingestion을 열 때 다시 판단한다.

### VerificationRequest

한 번의 확인 요청이다.

필드 후보:

- moment
- question
- requiredFields
- artifactIds
- savedFactIds

### TripFact

검토 가능한 여행 정보 후보다.

현재 공유 타입 이름은 `TripFact`다. 필드 후보:

- id
- schedule
- label
- value
- normalizedValue 또는 parsedValue
- evidenceState
- evidence
- reason
- sensitive
- conflictGroupId
- reviewDecision

### EvidenceRef

fact를 뒷받침하는 원문 근거다.

현재 공유 타입 이름은 `EvidenceRef`다. 필드 후보:

- artifactId
- label
- locator
- excerpt
- page
- region 또는 bbox
- extractionMethod

MVP에서는 자료명과 원문 발췌만으로 시작할 수 있다. page, region, bbox는 실제 파일/이미지 자료 인식이 필요해질 때 확장한다.

### EvidenceState

사용자가 다음 행동을 이해하기 위한 상태다.

현재 코드의 상태 이름은 `EvidenceState`다. 사용자 화면에서는 아래처럼 행동 언어로 번역한다.

| EvidenceState | 사용자 의미 |
| --- | --- |
| supported | 자료 근거로 확인됨 |
| needs_review | 근거는 있지만 사용자가 다시 봐야 함. 민감 정보는 이 상태와 `sensitive`로 직접 확인을 표현한다. |
| missing | 자료에 근거가 부족함 |
| conflict | 자료끼리 다른 값을 말함 |

### ReviewDecision

사용자의 검수 결과다.

현재 공유 타입에는 아직 `ReviewDecision`을 넣지 않는다. 우선 `src/client/`의 local state에서 저장, 수정, 무시를 표현하고, 서버나 eval이 같은 검수 결과를 읽어야 할 때 공유 타입으로 승격한다.

| Decision | 의미 |
| --- | --- |
| saved | 저장됨 |
| edited_and_saved | 수정 후 저장됨 |
| ignored | 무시됨 |
| marked_for_review | 직접 확인 대상으로 표시됨 |

### SituationCard

사용자가 최종적으로 보는 여행 순간별 확인 카드다.

현재는 `src/client/` 화면 구성에서 먼저 표현한다. 공유 타입으로 고정하기 전에 review UI와 eval 관찰 필요가 생기는지 본다.

규칙:

- 저장된 fact만 표시한다.
- 근거 부족 또는 충돌 상태는 사용자가 명시적으로 남긴 경우에만 경고나 확인 항목으로 표시한다.
- 민감 정보는 표시 수위를 조절한다.
