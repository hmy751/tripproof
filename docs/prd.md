# TripProof PRD

상태: 초안. 이 문서는 제품 요구사항을 정리하기 위한 기준 문서이며, 구현 완료나 평가 결과를 주장하지 않는다.

기준 문서: 제품의 단일 어휘·상태·흐름 기준은 `docs/product-model.md`다. 이 PRD는 기준 문서의 어휘·상태 2축·chat-first 흐름을 그대로 쓰고, 요구·acceptance·AI behavior를 그 위에서 정리한다. 개념·가치 정의가 기준 문서와 어긋나면 기준 문서를 따른다.

## Document Scope

이 문서는 TripProof의 첫 제품 요구사항을 사용자 문제, 사용자 흐름, 기능 요구, AI 동작 기준으로 정리한다. 기존 논의에서 나온 판단과 아이디어는 문서 안에서 작성 과정으로 설명하지 않고, 아래 기준에 맞춰 제품 요구사항으로 흡수한다.

이 PRD의 chat-first 흐름은 `docs/product-model.md`의 기준을 따른다. 자료함(Library)에 자료를 넣고 전체 자료함에 직접 묻는 채팅이 입구이며, 사용자가 근거를 보거나 직접 확인한 답변만 카드로 올려 일정 순서의 대시보드와 현장 카드로 정리한다. archive preview는 이 흐름을 정리할 때 참고한 과거 스냅샷이며, 현재 실행 기준은 아니다.

## 1. Product Overview

TripProof는 여행 자료함(Library)에 넣은 예약서, 바우처, 안내 메시지, 약관, 영수증, 캡처 같은 자료에 대해 AI와 묻고 답하면서, 사용자가 근거를 보거나 직접 확인한 답변만 카드로 올려 일정 순서의 대시보드와 현장 카드로 정리하는 여행 자료 확인 도구다.

대상 사용자는 여행 전후로 여러 자료를 받았고, 체크인, 투어 출발, 렌터카 반납, 결제 확인처럼 실수 비용이 큰 순간에 특정 정보를 빠르게 확인해야 하는 여행자다.

TripProof가 해결하려는 핵심 문제는 "자료를 찾기 어렵다"가 아니라 "필요한 정보가 어떤 자료에 근거하는지, 그 정보가 충분히 확정 가능한지, 자료끼리 충돌하지 않는지 사용자가 판단하기 어렵다"는 점이다.

TripProof는 일반 PDF 챗봇처럼 답변 로그만 쌓지 않는다. 그렇다고 AI가 자료를 읽자마자 대시보드를 자동 완성하지도 않는다. 채팅은 자료를 탐색하고 카드 후보를 발견하는 주 인터랙션이지만, 사람이 확인한 답변만 카드가 된다.

제품의 핵심 가치는 다음 네 가지다.

- 사용자가 전체 자료함에 직접 묻고, AI가 그 근거로 답한다.
- 답변마다 원문 근거를 답변 안에서 인라인으로 펼쳐 본다.
- 근거 있음, 확인 필요, 근거 부족, 자료 충돌 같은 근거 상태를 숨기지 않는다. 근거 부족·충돌이어도 사람이 직접 채워 올릴 수 있고, 그 카드는 직접 확인으로 정직하게 구분된다.
- 사용자가 확인한 카드만 대시보드에 오르고, 현장에서 다시 볼 카드만 현장 카드로 저장한다.

## 2. Problem Statement

여행자는 여행 준비와 이동 과정에서 여러 형태의 자료를 받는다.

- 호텔 예약 확인서
- 호스트 체크인 안내
- 투어 바우처
- 교통 티켓
- 렌터카 약관
- 보험 약관
- 결제 영수증
- 앱 화면이나 메시지 캡처

기존 방식의 한계는 다음과 같다.

- PDF 검색이나 메신저 검색은 사용자가 어떤 키워드로 찾아야 하는지 알고 있어야 한다.
- 일반 챗봇 답변은 답의 근거, 근거 부족, 자료 충돌을 충분히 드러내지 않을 수 있다.
- 문서 보관함은 자료를 모아두지만, 지금 행동에 필요한 정보와 확인 상태를 구분해주지 않는다.
- 사용자가 확인하지 않은 AI 추출값이 저장 정보처럼 보이면 잘못된 행동으로 이어질 수 있다.

TripProof가 다루는 문제 범위는 여행 자료함에 모인 자료에 대해 AI와 묻고 답하며, 답변에 근거와 상태를 함께 보여주고, 사용자의 확인을 거친 카드만 대시보드와 현장 카드로 만드는 흐름이다.

이번 PRD에서 다루지 않는 범위는 다음과 같다.

- 여행 일정 자동 생성
- 항공권, 숙소, 투어 예약 또는 결제
- 모든 여행 자료를 저장하는 범용 문서 금고
- 문서 하나만 고르는 PDF 챗봇 모드
- 제품 품질 점검용 대시보드
- 다중 사용자, 권한 관리, 배포 운영

## 3. Goals / Non-goals

### Goals

- 사용자가 이번 여행 자료(Artifact)를 자료함(Library)에 넣고, 전체 자료함을 대상으로 직접 물을 수 있다.
- 자료를 넣으면 AI가 먼저 확인해볼 후보(AICandidate)를 제안한다.
- 채팅 답변(ChatAnswer)은 항상 근거 상태(EvidenceState)를 가지고, 근거가 있으면 답변 안에서 인라인으로 펼쳐 본다.
- 근거가 부족한 항목은 답을 만들어내지 않고 `근거 부족` 상태로 멈추되, 사람이 직접 채워 올리는 것을 막지 않는다.
- 서로 다른 자료가 같은 항목을 다르게 말하면 하나로 합치지 않고 `자료 충돌` 상태로 양쪽 근거를 함께 보여준다.
- 사용자가 답변·후보를 카드 초안(CardDraft)으로 올려 확인·편집할 수 있다.
- 사용자가 확정한 카드만 대시보드(Dashboard)에 일정 순서로 오르고, 현장에서 다시 볼 카드만 현장 카드(FieldCard)로 저장된다.
- 제품 품질은 답이 자료와 맞는지, 근거가 정확한지, 근거 부족에서 멈추는지, 자료 충돌을 드러내는지의 관점에서 점검 가능해야 한다.

### Non-goals

Non-goal은 영구 금지가 아니다. 아래 첫 묶음은 **추후 확장할 범위**(로드맵에 있고 이번 버전이 거기까지 가지 않을 뿐)이고, 둘째 묶음은 버전과 무관하게 **유지하는 정직성 원칙**이다.

추후 확장할 범위 (로드맵 · 이번 버전 다음):

- 여행 도메인 확장 — 투어·렌터카·결제 등은 MVP contract가 안정되면 카테고리로 넓힌다. 이번 버전은 숙소 체크인에 집중한다(§6 Later Features).
- 자료 형태 확장 — 실제 파일·이미지 자료 인식과 이미지 영역 근거 표시는 자료 형태 수요가 확인되면 연다. 이번 버전은 텍스트 자료로 먼저 검증한다(§6 Later Features).
- 원자료 속 식별성 값의 별도 처리 — 이번 P0은 답변 상태, 근거, 사람 확인 흐름에 집중한다. 원자료 속 특정 값을 별도로 탐지·가림·보존하는 기능은 실제 공유·운영 기능을 열 때 별도 product contract로 다룬다.

버전과 무관하게 유지하는 원칙 (정직성 계약, 버전이 올라가도 약화하지 않는다):

- AI가 추출한 정보를 사용자 확인 없이 카드나 대시보드에 confirmed 정보로 올리지 않는다.
- 품질 점검 지표·실행 기록·내부 진단 정보를 사용자 화면의 중심 언어로 쓰지 않는다.
- AI confidence 수치를 사용자 언어로 쓰지 않는다.
- 공개용 품질 수치나 개선 전후 상태를 실제 실행 전 문서에 쓰지 않는다.

### MVP 기준

MVP는 "여행 자료 확인 도구"의 전체 카탈로그가 아니라, 한 사용자 순간을 chat-first 흐름으로 끝까지 통과시키는 얇은 흐름이다.

MVP는 도메인을 좁히는 것이지 인터랙션을 미루는 것이 아니다. 첫 MVP 장면은 숙소 체크인이라는 도메인 하나로 범위를 좁히지만, chat-first(전체 자료함 채팅 → 답변 상태·인라인 근거 → 카드 초안 → 대시보드 → 현장 카드)는 핵심 인터랙션으로 함께 통과시킨다. 채팅을 나중 기능으로 미루지 않는다.

MVP 시나리오는 숙소 체크인 확인이다. P0에서는 예약 확인서와 호스트 안내 같은 자료를 자료함에 넣고, 전체 자료함에 물어 체크인 시작 시간과 늦은 도착 조건을 확인한다. 추가 제시물 필요 여부와 출입 방법 전체 추출은 후속 확장으로 둔다.

이 시나리오는 최종 제품 전체를 대표하지는 않지만, TripProof의 핵심 요구사항인 전체 자료함 채팅, 답변 상태와 인라인 근거, 근거 부족·충돌 표현, 사람 확인 게이트, 사용자가 올린 카드만 대시보드 반영을 작게 검증하기에 적합하다.

## 4. Target User & Use Cases

### Target User

주요 사용자는 직접 여행 자료를 관리하고, 여행 중 중요한 조건을 스스로 확인해야 하는 개인 여행자다.

이 사용자는 다음 상황에 자주 놓인다.

- 이동 중이라 긴 문서를 다시 읽기 어렵다.
- 여러 앱, 이메일, PDF, 메시지에 정보가 흩어져 있다.
- 잘못 알면 체크인 실패, 투어 지각, 추가 결제, 보증금 문제 같은 비용이 생긴다.
- AI에게 물어보고 싶지만 근거 없는 답은 믿기 어렵다.
- AI가 근거를 못 찾아도, 본인이 원문에서 확인한 내용은 직접 채워 남기고 싶다.

### Use Cases

| Use case | 사용자 상황 | 확인할 정보 |
| --- | --- | --- |
| 숙소 체크인 | 숙소 근처에 도착했거나 이동 중이다. | 체크인 시간, 늦은 도착 조건, 추가 제시물 필요 여부, 출입 방법 |
| 투어 출발 | 투어 시작 직전 바우처와 안내 메시지를 확인한다. | 집합 시간, 집합 장소, 모바일 바우처 가능 여부, 지각 규정 |
| 렌터카 반납 | 차량 반납 전 약관과 예약 조건을 확인한다. | 연료 정책, 보증금 조건, 반납 위치, 반납 시간 |
| 여행 후 결제 확인 | 여행 후 영수증과 예약 조건을 대조한다. | 추가 결제 이유, 도시세, 보증금 환불 조건, 취소/환불 규정 |

이 use case들은 대시보드에서 카드의 카테고리(숙소·투어·렌터카·결제) 분류로도 이어진다. MVP는 위 use case 중 숙소 체크인을 먼저 깊게 닫는다. 나머지는 같은 chat-first 구조를 확장할 후보로 둔다.

## 5. Core User Flow

핵심 흐름은 chat-first다. 사용자는 '여행 순간'을 먼저 고르지 않는다. 자료를 자료함에 넣고, 전체 자료함에 직접 묻는 것이 입구다. 자세한 흐름 정의와 화면 위계(탭)는 기준 문서(`docs/product-model.md`)를 따른다.

```text
자료함 (Library)
  → AI 추천 후보(AICandidate) + 전체 자료함 채팅(Library Chat)
  → 답변(ChatAnswer: 근거 축 상태 + 인라인 근거)
  → 카드 초안(CardDraft: 사람 확인·편집, 근거 부족해도 직접 채워 올리기 허용)
  → 대시보드(Dashboard: 일정 → 카테고리)
  → 현장 카드(FieldCard)
```

### MVP Flow

1. 사용자가 이번 여행 자료(Artifact)를 자료함(Library)에 첨부한다.
   - 예: `호텔 예약 확인서`, `호스트 체크인 안내`, `앱 메시지 캡처`
   - 자료함에는 원본이 보존되고, 이후 근거 표시에서 자료명으로 식별된다.
2. 자료를 넣으면 AI가 먼저 확인해볼 후보(AICandidate)를 화면 오른쪽 rail에 제안한다.
   - 후보는 대시보드 카드가 아니라 검토 전 제안이다. 채팅으로 보내거나 바로 카드 초안으로 보낼 수 있다.
3. 사용자가 전체 자료함을 대상으로 직접 묻는다.
   - 예: `호텔에 늦게 도착하면 어떻게 돼?`
   - 문서 하나를 고르는 PDF 챗봇이 아니다. 기본 대상은 자료함 전체다.
4. AI가 전체 자료를 근거로 답하고, 답변(ChatAnswer)에 근거 상태를 붙인다.
   - `근거 있음` / `확인 필요` / `근거 부족` / `자료 충돌`
5. 근거가 있으면 답변 안에서 `근거 N개 펼치기`로 인라인 확인한다.
   - 근거가 없으면 "AI가 못 찾았을 수 있으니 직접 확인이 필요하다"를 명시한다. 답변은 자동으로 카드가 되지 않는다.
6. 사용자가 답변이나 후보를 카드 초안(CardDraft)으로 올린다.
   - 채팅 아래에 초안이 뜬다. 카테고리·일정·카드 이름·값을 확인·편집한다.
   - 근거 부족·충돌이어도 막지 않는다. 사람이 값을 직접 채워 올릴 수 있다.
   - supported였어도 사람이 값·이름을 고치면 출처가 `직접 확인`으로 바뀐다.
7. 초안을 확정하면 대시보드(Dashboard)에 카드(DashboardCard)로 오른다.
   - 대시보드는 사람이 올린 카드만 담고, 일정(타임라인) → 카테고리로 묶는다.
   - AI 근거 카드(`근거 있음`)와 직접 확인 카드(`직접 확인`)를 구분해 보여준다.
8. 사용자가 대시보드에서 `현장 저장`을 누른 카드만 현장 카드(FieldCard)로 모인다.
   - 일정별로 크게, 현장에서 빠르게 다시 볼 정보의 도착점이다. 사람 저장 없이 자동으로 들어가지 않는다.

### Expected Result

사용자는 긴 자료 전체를 다시 읽지 않고도, 전체 자료함에 직접 물어 지금 행동에 필요한 정보와 그 근거를 확인하고, 확인한 정보만 일정 순서의 대시보드와 현장 카드로 정리할 수 있어야 한다. 근거가 부족하거나 충돌하는 항목도 직접 확인해 올릴 수 있고, 그 카드는 정직하게 구분된다.

## 7. Acceptance Criteria

### 자료 입력 / Artifact Input

- 사용자는 최소 두 개 이상의 자료를 이름과 함께 자료함에 추가할 수 있다.
- 각 자료는 이후 근거 표시에서 자료명으로 식별 가능해야 한다.
- 자료명이 없는 자료는 사용자가 구분 가능한 기본 이름을 받아야 한다.

### Chat Answer

- 시스템은 전체 자료함을 대상으로 한 질문에 답하고, 답변마다 근거 상태(EvidenceState)를 표시해야 한다.
- 답변은 근거 없이 확정형으로 끝나지 않는다.
- 자료에 없는 값은 일반 지식으로 채우지 않는다.
- 자료에 있는 표현이 애매하면 `확인 필요` 상태로 둔다.

### Evidence

- `근거 있음`(supported) 답변·후보·카드는 최소 하나 이상의 원문 근거(EvidenceRef)를 가져야 한다.
- 근거는 사용자가 원자료에서 다시 찾을 수 있을 만큼 구체적이어야 한다.
- 근거는 답변 안에서 인라인으로 펼쳐 본다. 결정과 분리된 채 별도 패널로 떠다니지 않는다.
- 답변에 쓰인 값과 원문 발췌가 서로 맞지 않으면 완료로 보지 않는다.

### Insufficient Evidence

- 사용자가 묻는 정보가 자료에 없으면 시스템은 값을 만들어내지 않는다.
- 이 경우 답변은 `근거 부족`으로 표시되고, 왜 답할 수 없는지 짧게 설명한다. AI가 못 찾았을 수 있음을 함께 알린다.
- `근거 부족`은 카드 올리기를 막지 않는다. 사용자가 직접 값을 채워 올리면 그 카드는 `직접 확인` 출처로 대시보드에 오른다.

### Conflict

- 같은 항목에 대해 서로 다른 자료가 다른 값을 말하면 시스템은 하나를 선택하지 않는다.
- 충돌한 값과 각 값의 근거가 함께 표시되어야 한다.
- `자료 충돌` 답변도 카드 올리기를 막지 않는다. 사용자가 직접 확인해 값을 채워 올리면 `직접 확인` 카드가 된다.

### Card Draft / Dashboard

- 사용자는 답변·후보를 카드 초안으로 올려 카테고리·일정·이름·값을 편집할 수 있다.
- 사용자가 초안을 확정한 카드만 대시보드에 표시된다.
- 무시한 답변·후보는 대시보드에 나타나지 않는다.
- AI의 supported 답변을 값·이름 수정 없이 올린 카드(`근거 있음`)와 사람이 직접 채운 카드(`직접 확인`)는 구분 가능해야 한다.
- supported 답변이라도 사람이 값·이름을 고치면 그 카드는 `직접 확인`으로 표시된다.

### Field Card

- 현장 카드에는 사용자가 대시보드에서 `현장 저장`으로 저장한 카드만 들어간다.
- 사용자가 저장하지 않은 카드가 현장 카드에 자동으로 들어가면 안 된다.

### Examples And Fixtures

- 예시, sample 자료, 품질 리포트는 제품 기능 완료 기준이 아니다.
- 원문 일부를 화면에 얼마나 보여줄지, 기록에 무엇을 남길지는 P0 숙소 체크인 구현 조건이 아니다. 실제 자료 입력과 demo 수위가 정해진 뒤 별도 판단한다.

### AI Failure Cases

- 근거 없는 답변을 확정형으로 생성하는 것은 실패다.
- 근거 위치가 실제 답의 근거를 가리키지 않으면 실패다.
- 자료 충돌을 단일 답으로 뭉개면 실패다.
- 근거 부족이라고 사용자의 카드 올리기를 막으면 실패다(AI 판단이 사용자를 막아서는 안 된다).
- 직접 확인 카드와 AI 근거 카드가 구분되지 않으면 실패다.
- AI confidence 수치나 내부 평가 용어만 노출되어 사용자가 다음 행동을 이해하지 못하면 실패다.

## 8. AI Behavior Requirements

### AI Inputs

AI는 다음 입력을 기준으로 동작해야 한다.

- 사용자가 자료함에 넣은 자료 묶음
- 각 자료의 이름
- 사용자의 질문(전체 자료함 대상)
- 이미 대시보드에 오른 카드가 있다면 그 목록
- 사용자가 직접 수정하거나 채운 값

### AI Outputs

AI는 자유 답변보다 검토 가능한 구조를 우선 출력해야 한다. 아래는 제품 요구상 필요한 출력 항목이다.

채팅 답변(ChatAnswer)의 각 항목은 다음을 포함한다. AI 내부 구현은 extraction 후보나 RAG 후보를 만들 수 있지만, 현재 사용자-facing 출력은 ChatAnswer를 기준으로 설명한다.

- 항목 이름(label)
- 항목 값 또는 값이 없음을 나타내는 상태
- 근거 상태(evidenceState). 사용자에게는 근거 상태 문구(근거 있음/확인 필요/근거 부족/자료 충돌)로 보인다.
- 근거 목록(evidence)
- 왜 해당 상태로 판단했는지 설명하는 짧은 reason
- conflict 관계가 있다면 충돌한 후보 목록과 각 후보의 근거

### Uncertainty Handling

AI는 불확실한 정보를 확정값처럼 말하면 안 된다.

- 자료에 근거가 없으면 `근거 부족`(missing)으로 멈추고, AI가 못 찾았을 수 있음을 함께 알린다.
- 자료의 표현이 조건부이면 조건을 함께 표시한다.
- 날짜, 시간, timezone, 금액, 환불 조건이 애매하면 `확인 필요`(needs_review)로 둔다.
- 여러 자료가 충돌하면 `자료 충돌`(conflict)로 양쪽 근거를 함께 보여준다.
- 근거 상태는 사람이 카드로 올릴 수 있는지를 막지 않는다. 카드 올리기 여부와 출처는 사람이 결정한다.

### User Confirmation Required

다음 상황에서는 사용자 확인이 필요하다(사람 확인 게이트).

- 자료끼리 같은 항목을 다르게 말하는 경우
- 근거는 있지만 조건이 복잡하거나 예외가 있는 경우
- AI가 값은 답했지만 근거 위치가 약한 경우
- 사용자가 직접 수정하거나 채운 값이 기존 근거와 다를 경우

### AI Must Not

- 자료 밖 일반 지식으로 여행 조건을 채우지 않는다.
- 두 자료의 충돌을 임의로 해결하지 않는다.
- 사용자가 확정하지 않은 답변·후보를 대시보드나 현장 카드에 confirmed 정보처럼 표시하지 않는다.
- 근거 부족·충돌을 이유로 사용자의 카드 올리기를 막지 않는다.
- AI confidence 수치('AI confidence 0.84', '신뢰도 medium')를 사용자 언어로 쓰지 않는다.
- 내부 진단 용어나 구현 타입명을 사용자 설명의 중심에 두지 않는다.
- 구현되지 않은 자료 인식, 충돌 분석, 품질 점검 결과를 완료된 기능처럼 표현하지 않는다.

## 9. Data / Domain Model

이 섹션은 구현 언어의 확정 타입이 아니라 제품 요구를 설명하기 위한 도메인 모델이다. 객체 모델과 상태 2축의 단일 정의는 기준 문서(`docs/product-model.md`)를 따르며, 여기서는 PRD에 필요한 만큼만 정리한다.

### 객체 모델

| 객체 | 역할 | 제품 요구 |
| --- | --- | --- |
| Trip | 확인하려는 하나의 여행 묶음 | 자료, 질문, 카드, 대시보드가 같은 여행 맥락 안에서 이어져야 한다. |
| Artifact | 자료함에 넣은 원본 자료 | 답변과 근거가 다시 찾을 수 있는 원천이다. |
| Library (자료함) | Artifact를 모으고 근거의 원천이 되는 장소 | 사용자는 전체 자료함을 대상으로 묻는다. |
| Library Chat | 전체 자료함에 묻는 주 인터랙션 | 답변과 카드 후보를 발견하는 주 입구다. |
| ChatAnswer | 상태와 인라인 근거가 붙은 답변 | 근거 상태, reason, 근거 목록을 함께 가진다. |
| AICandidate | 자료를 넣자마자 AI가 제안하는 확인 후보 | 확정 정보가 아니라 검토 전 제안이다. |
| CardDraft | 카드가 되기 전 사람이 확인·편집하는 단계 | 답변이나 후보가 자동 확정되지 않게 막는다. |
| DashboardCard | 사람이 확정해 대시보드에 올린 카드 | 사람이 확인한 정보만 담는다. |
| FieldCard | 현장에서 다시 보기로 저장한 카드 | 사용자가 현장 저장한 카드만 모인다. |
| Evidence | 답변·후보·카드가 가리키는 원문 근거 | 사용자가 원자료에서 다시 찾을 수 있을 만큼 구체적이어야 한다. |
| 일정 (Schedule) | 카드의 시간축 분류 | 여행 중 언제 봐야 하는 정보인지 정리한다. |
| 카테고리 (Category) | 카드의 종류 분류 | 숙소·투어·렌터카·결제처럼 서로 다른 업무를 빠르게 스캔하게 한다. |

### 상태 2축

상태는 한 축이 아니라 두 직교 축이다. 근거 축(EvidenceState)은 "AI가 자료로 이 답을 댈 수 있었는가"라는 답변 레벨 질문이고, 결정 축(ReviewDecision / 카드 출처)은 "사람이 이 정보를 카드로 어떻게 올렸는가"라는 카드 레벨 질문이다. 근거 축은 카드로 올릴 수 있는지를 막지 않는다.

#### 근거 축 — EvidenceState (답변 레벨)

제품 상태 이름은 `EvidenceState`이고, 답변과 후보에 붙는다.

| EvidenceState | 사용자 문구 | 의미 | 붙는 곳 |
| --- | --- | --- | --- |
| `supported` | 근거 있음 | 현재 자료 근거로 답할 수 있음(근거 1개 이상) | 답변 · 후보 |
| `needs_review` | 확인 필요 | 근거가 있어도 애매해 원문 확인이 필요함 | 답변 · 후보 |
| `missing` | 근거 부족 | 현재 자료로는 AI가 답을 못 찾음(못 찾았을 뿐일 수 있음) | 답변 · 후보 |
| `conflict` | 자료 충돌 | 서로 다른 자료가 같은 항목에 다른 값을 말함 | 답변 · 후보 |

#### 결정 축 — ReviewDecision / 카드 출처 (카드 레벨)

결정 축(카드 출처와 supported→직접 확인 전환 규칙 포함)은 카드 레벨 상태다. 답변의 근거 상태와 섞지 않는다.

| 카드 출처 | 사용자 문구 | 의미 | 붙는 곳 |
| --- | --- | --- | --- |
| supported origin | 근거 있음 | AI의 supported 답변을 값·이름 수정 없이 그대로 올린 카드(별도 출처 라벨을 새로 붙이지 않고 근거 축 supported를 그대로 표시) | 대시보드 카드 |
| user (직접 확인) | 직접 확인 | 사람이 원문에서 확인한 값을 채워 올린 카드. 근거가 부족/충돌이었거나, supported였어도 사람이 값을 고치면 이 출처로 바뀐다. | 대시보드 카드 |
| ignored | (올리지 않음) | 답변·후보를 카드로 올리지 않기로 한 결정 | 대시보드에 없음 |
| saved (현장 저장) | 현장 저장 | 대시보드 카드를 현장에서 다시 보기로 저장한 결정(overlay) | 카드 overlay → 현장 카드 |

#### 어휘 분리 (EvidenceState ⟂ ReviewDecision)

- `확인 필요`는 근거 축의 `needs_review`(답변 레벨)다. "AI는 답했지만 네가 원문을 다시 봐"라는 뜻이다.
- `직접 확인`은 결정 축의 `user`(카드 레벨)다. "네가 직접 원문에서 확인해 채워 올린 카드"라는 뜻이다.
- 두 개념을 한 단어로 섞지 않는다. 근거 부족(`missing`) 답변도 사람이 값을 채우면 `직접 확인` 카드로 올라간다.
- `근거 있음`은 근거 축과 결정 축에 같은 단어로 나타나지만(supported 답변이 수정 없이 카드까지 보존된 경우), 사람이 값·이름을 고치면 `직접 확인`으로 바뀌므로 직교성을 깨지 않는다.

### 도메인 계약

문서의 제품 도메인 이름은 유지한다: `EvidenceState`, `Artifact`, `EvidenceRef`, `ChatAnswer`, `CardDraft`, `DashboardCard`.

- `EvidenceRef`는 원문 근거로 되돌아가기 위한 참조다. 현재 API/schema에서는 `materialId`, `sourceUnitId`, `label`, `locator`, `snippet`을 가진다. 제품 언어의 Artifact를 구현 record로 다룰 때는 Material / SourceUnit 경계를 거친다. SourceUnit은 검색·grounding 단위이며, 사용자-facing Evidence 자체가 아니다.
- `ChatAnswer`의 항목은 `id`, `label`, `body`, `value`, `evidenceState`, `evidence`를 가진다.
- `CardDraft`와 `DashboardCard`는 사람이 카드로 올리는 결정 축을 가진다. 현재 client 구현은 `sourceKind: "evidence" | "manual"`로 AI 근거 카드와 직접 확인 카드를 구분한다.
- `FactCandidate` 같은 추출 후보 이름은 내부 후보 생성 용어다. 현재 사용자-facing 답변과 카드 경로는 `ChatAnswer -> CardDraft -> DashboardCard`를 기준으로 쓴다.

구현 계약으로 옮길 때 지켜야 할 점:

- `missing`이나 `conflict`는 값이 없거나 여러 값이 있을 수 있다.
- 대시보드에서 스캔해야 하는 정보는 일정과 카테고리로 분류할 수 있어야 한다.
- 결정 축(카드 출처)은 근거 축과 분리해야 한다.
- conflict는 충돌한 값과 각 근거를 함께 보존해야 한다.
- AI confidence 수치는 사용자 언어로 쓰지 않는다.

### 정직성 원칙

- 근거 없는 답을 만들어내지 않는다. 자료에 없으면 `근거 부족`으로 멈추고 왜 답할 수 없는지 짧게 설명한다.
- 서로 다른 자료의 값을 하나로 뭉개지 않는다. `자료 충돌`로 양쪽 근거를 함께 보여준다.
- 사람이 확정하지 않은 답변·후보는 대시보드나 현장 카드에 confirmed 정보처럼 표시하지 않는다.
- 근거 부족·충돌이어도 카드 올리기를 막지 않는다. AI가 못 찾았을 수 있으므로 사람이 직접 채워 올릴 수 있고, 그 카드는 `직접 확인`으로 정직하게 구분한다.
- 근거 있음(`supported`) 보증은 사람이 값·이름을 고치면 무효이므로, 그 카드의 출처를 `직접 확인`으로 바꿔 AI 근거 카드와 섞이지 않게 한다.
- AI confidence 수치를 사용자 언어로 쓰지 않는다. 사용자 문구는 근거 있음 / 확인 필요 / 근거 부족 / 자료 충돌 / 직접 확인 / 현장 저장만 쓴다.
- 내부 평가 용어나 구현 타입명을 사용자 설명의 중심에 두지 않는다.
