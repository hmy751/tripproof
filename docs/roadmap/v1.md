# TripProof 1차 로드맵 · MVP 계획

상태: **1차 로드맵 (느슨한 후보 메뉴).** 고정된 계획이 아니라 언제든 갈아엎고 다음 버전으로 다시 쓸 수 있는 출발점이다. 아직 구현된 product flow / eval run / before-after 결과를 뜻하지 않는다. 여기 적힌 MVP·확장 항목은 "도달하려는 목표"이고, 현재 코드 상태는 아래 "현재 상태(출발점)"에 따로 적는다.

## 이 문서의 역할 (먼저 읽기)

이 문서는 **다른 세션이 혼자 작업 후보를 이해할 수 있게** 하는 메뉴다. 일정표도, 단계 계약도, 현재 실행 명령도 아니다.

- **세션을 묶지 않는다.** "지금 N단계 차례" 같은 강한 순서·날짜를 두지 않는다. 실제 선택은 사용자 요청과 현재 코드 상태에서 닫는다.
- **기준은 다른 문서에 있다.** 어휘·상태·흐름은 `product-model.md`, 요구·경계는 `prd.md`, 첫 slice 계약은 `specs/accommodation-checkin.md`, 유지 기준은 `specs/00-spec-driven-development.md`. 이 문서는 그걸 **재서술하지 않고**, "무엇을 어떤 순서 감각으로 열까"만 다룬다.

작업을 고를 때 판단 질문:

- 이 항목이 사용자 흐름을 끝까지 통과시키는가, 아니면 보조층(fixture/eval/도구)을 앞세우는가?
- 지금 코드 상태에서 열 수 있는가, 막은 선행 항목이 있는가?
- 끝나면 무엇이 "참"이 되는지(통과 기준)를 한 줄로 말할 수 있는가?

## 변하지 않는 목표

여행 자료함에 묻고, 근거를 보거나 직접 확인한 답변만 카드로 올려 일정 대시보드·현장 카드로 정리하는 여행 자료 확인 도구. 핵심 가치는 **믿어도 되는 정보와 아직 직접 확인할 정보를 근거와 함께 정직하게 분리**하는 것. (정의 원문은 `product-model.md` §1)

## 현재 상태 (출발점 — 2026-05-30 기준)

문서는 chat-first로 정렬됐지만 코드는 아직 거기 안 닿아 있다. 다른 세션이 알아야 할 출발점:

- **실행 세팅 없음.** root에 `package.json`·`tsconfig`·번들러 없음. `src/server/ai/*.ts`는 빌드도 실행도 안 되는 파일 상태.
- **`src/client/app.js` = 옛 moment-first 데모.** 문서가 버린 "여행 순간" 그리드, 단일 상태 `pending/approved/rejected`, confidence 바 노출, 키워드 매칭 가짜 답변. chat-first 아님.
- **`src/server/ai/`는 새 모델에 가까우나 미연결.** `normalizeTripFacts.ts`는 진짜 `EvidenceState` 로직(conflict/missing/needs_review/supported + openIssues)을 갖지만, `extractTripFacts.ts`는 하드코딩 stub이고 아무도 호출하지 않는다.
- **client·server가 각자 다른 하드코딩 데이터로 분리됨.** `src/shared/tripFacts.ts` 계약을 client가 import하지 않는다 (같은 데이터 2벌).

→ 그래서 진짜 일은 "폴더 옮기기"가 아니라 **chat-first 웹 재작성 + 실제 계약 연결 + 실행 가능화**다. 폴더 재편은 그 그릇일 뿐이다.

## 후보: 구조 정비

목적: MVP가 통과할 그릇을 만들고 AI를 상위 관심사로 올린다. **지금 코드량(~1.5k줄, 빌드도 없음)에 완성형 모노레포를 미리 깔지 않는다.** 필요 최소만.

이 후보가 열릴 때 확인할 수 있는 것:

- AI 코드 위치를 현재 `src/server/ai/`에 둘지, 상위 관심사로 올릴지.
- 최소 실행 세팅 1벌이 필요한지.
- client·server 데이터 2벌을 `shared` 계약 1벌로 모으는 연결점이 필요한지.

구조 선택지는 실제 작업을 열 때 다시 판단한다. 단일 패키지 + 경계 별칭, 폴더만 정리, 처음부터 워크스페이스 같은 선택지가 있었지만 이 문서는 그중 하나를 현재 명령으로 고정하지 않는다. 실제 구조를 바꾸면 `decisions/`에 남긴다.

⚠️ 이 항목은 `00-spec-driven-development.md`가 적은 현재 구조 베이스라인(AI = `src/server/ai`)을 바꾼다. 실제로 옮기면 `decisions/`에 결정 노트를 남기고 `README.md`·`00-spec-driven-development.md`의 구조 색인을 같이 맞춘다. 단 유지 기준은 그대로 지킨다 — 의존 방향 `eval → src`, `src/product` wrapper는 만들지 않음.

끝나면 참이 되는 것: 브라우저에서 화면이 실제 `shared` 타입을 import해 한 번 렌더되고, AI 관련 모듈을 UI가 계약으로 호출할 수 있다.

## 후보: MVP가 끝까지 통과시킬 한 흐름

한 문장: **숙소 체크인 흐름이 가짜 2벌이 아니라 실제 `shared` 계약을 통해 브라우저에서 끝까지 돈다.**

시나리오는 새로 만들지 않는다. `specs/accommodation-checkin.md`의 그 시나리오(체크인 시작 시간 + 늦은 도착 조건, 필드 2개)를 MVP가 끝까지 통과시킬 한 흐름으로 고정한다. 이걸 고른 이유: 도메인은 작은데 chat-first 흐름(자료함 → 채팅 → 답변+근거 → 카드 초안 → 대시보드 → 현장)을 끝까지 통과시키고, 한 필드는 `근거 있음`·다른 하나는 `근거 부족`/`확인 필요`로 갈려, 잘 풀리는 경우만 보여주는 게 아니라 핵심 가치(믿을 정보와 직접 확인할 정보를 정직하게 나눔)를 보여준다.

끝나면 참이 되는 것:

- 자료함에 synthetic fixture 2개(예약 확인서 + 호스트 안내)를 넣을 수 있다.
- 확인(채팅) 탭에서 전체 자료함에 "체크인 몇 시부터?", "늦게 도착하면?"을 물으면, 답변에 근거 축 상태(근거 있음/확인 필요/근거 부족/자료 충돌)와 인라인 근거가 붙는다.
- 두 질문이 같은 상태로만 끝나지 않는다 — 하나는 `근거 있음`, 다른 하나는 `근거 부족`/`확인 필요`로 갈려 정직한 분리를 보여준다.
- `근거 부족` 건을 사용자가 원문을 보고 직접 채워 올리면 카드 출처가 `직접 확인`이 된다.
- 올린 카드만 일정 → 카테고리 대시보드에 뜨고, 현장 저장한 카드만 현장 탭에 모인다.
- confidence 수치·바를 사용자 화면에 노출하지 않는다.
- 데이터가 하드코딩 2벌이 아니라 `shared` 계약 1벌을 지난다.

MVP에 넣지 않음 (`specs/accommodation-checkin.md` 제외 그대로): 실제 PDF·이미지·이메일 ingestion, 진짜 LLM 호출, OCR, 일반 open-ended Q&A, metric 계산. 단 민감값(예약번호·출입 코드·연락처) 자동 저장 방지 guard는 포함한다.

작업 감각: product behavior를 먼저 통과시키고, test/eval은 그 뒤에 결과를 호출해 관찰한다. 세부 구현 순서는 현재 작업 brief에서 닫는다.

## 후보 목록: 후속 확장

MVP가 돌기 시작하면 아래에서 막힌 것 없는 항목을 고를 수 있다. 날짜·강한 순서 없음. 각 항목은 "여는 순간 / 되는 것"만 적는다.

- **LLM adapter** — deterministic baseline이 답할 수 있는 한계가 보이면 연다(유지 기준: 큰 후보는 baseline 실패가 보일 때). `src/ai`에 provider 경계를 두고 server가 secret을 숨긴다.
- **실제 자료 ingestion (PDF·이미지·메시지)** — synthetic text로 흐름이 안정된 뒤. `TravelArtifact.kind`별 텍스트 추출부터.
- **충돌 UX 완성 (conflict)** — 한 항목에 두 값이 보일 때 양쪽 근거를 나란히 보여준다. 출력 계약의 `conflictCandidates` 확장 후보와 연결.
- **계약 확장** — `value: string | null`, `category`, 카드 출처/`ReviewDecision`의 `shared` 승격. 필요해지는 항목부터 작게. (`product-model.md` §3 확장 후보)
- **둘째 도메인 (투어 / 렌터카 / 결제)** — 같은 contract로 카테고리만 넓힌다. 이때 구조를 워크스페이스 모노레포로 승격할지 판단한다.
- **eval observer** — product entry point가 돌고 baseline을 관찰할 수 있을 때. 축 이름은 유지한다(Faithfulness/Groundedness, Citation Precision, Abstention F1, Conflict Recall). product가 eval에 의존하지 않게 둔다.

이 목록은 닫아두지 않는다 — 새 후보가 보이면 한 줄로 추가한다.

## 사용 메모

이 문서는 현재 작업 지시가 아니다. 후보 선택은 사용자 요청과 현재 코드 상태에서 닫는다.
