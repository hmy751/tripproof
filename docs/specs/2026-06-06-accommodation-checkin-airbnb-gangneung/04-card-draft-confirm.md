# 숙소 체크인 04 - 카드 초안과 직접 확인

상태: sub-spec draft.

부모: [숙소 체크인 확인 - Agoda 후쿠오카 예약 PDF](index.md)

## 왜 지금

TripProof는 답변 로그만 쌓는 도구가 아니다. 사용자가 남길 정보를 카드 초안으로 올리고, AI가 PDF에서 못 찾은 값도 직접 확인으로 채울 수 있어야 한다. 이번 단계는 채팅 답변에서 카드 초안으로 넘어가는 흐름을 닫는다.

## 사용자 장면

사용자는 체크인 답변을 보고 `예약 확정서 제시` 안내는 카드로 남기고 싶다. 체크인 시작 시각은 PDF에 없지만, Agoda 메일 본문이나 숙소 앱에서 확인한 뒤 직접 적어두고 싶다.

## Goal

- 체크인 시 예약 확정서 제시 답변을 카드 초안으로 올릴 수 있다.
- 카드 초안에는 일정, 이름, 값, 상태, 출처가 보인다.
- 사용자가 초안의 이름이나 값을 고칠 수 있다.
- 체크인 시작 시각처럼 `근거 부족`인 항목도 직접 값을 입력할 수 있다.
- 직접 입력하거나 수정한 카드는 `직접 확인`으로 구분된다.
- 예약번호, 고객명, 결제 카드, 숙소 연락처, 정확한 주소는 자동 초안으로 올라가지 않는다.
- 카드 초안은 대시보드에 자동 확정되지 않는다.

## Rules

- `DraftListPanel`은 빈 화면이 아니라 초안 목록과 편집 상태를 가진다.
- draft state는 채팅 답변의 항목에서 만들어진다.
- `근거 있음` 초안과 `직접 확인` 초안은 출처 표시가 다르다.
- 직접 입력값은 PDF 근거 snippet처럼 보이지 않게 한다.
- 민감 필드 후보는 초안 생성 전에 막거나, 생성되더라도 명시적인 확인 상태로 격리한다.

## Non-goals

- Agoda나 숙소에 메시지를 보내는 기능.
- 모든 카드 편집 UX.
- 대시보드 배치는 05에서 다룬다.
- 민감정보 표시 세부 정책 완성.
- companion source에서 자동으로 직접 확인 값을 채우기.

## 현재 코드에서 볼 곳

- `src/client/components/DraftListPanel.tsx`: 현재 항상 빈 초안이다.
- `src/client/App.tsx`: draft state가 아직 없다.
- 기존 `src/shared/tripFacts.ts`: 답변 상태와 카드 출처를 어떻게 나눌지 확인이 필요했다. Python backend 전환 후에는 backend response schema와 client draft state의 경계로 다시 본다.

## 이번 AC

1. 체크인 제시물 답변에서 카드 초안을 만들 수 있다.
2. 초안에서 값과 이름을 확인하거나 수정할 수 있다.
3. `근거 부족`인 체크인 시작 시각도 직접 값을 채울 수 있다.
4. 직접 채운 카드는 `직접 확인`으로 표시된다.
5. 예약번호, 고객명, 결제 카드 같은 민감 필드는 자동 초안이 되지 않는다.
6. 초안은 사용자가 확정하기 전까지 대시보드에 보이지 않는다.

## 남은 판단

- `근거 부족` 답변에서 바로 빈 초안을 만들지, 사용자가 먼저 `직접 확인으로 남기기`를 눌러야 할지.
- 카드 출처를 backend response schema에 둘지, client draft state에서만 둘지.
- 민감정보 guard를 카드 후보 생성 전 단계에 둘지, 초안 UI 단계에 둘지.
