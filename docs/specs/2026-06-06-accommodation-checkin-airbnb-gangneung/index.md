# 숙소 체크인 확인 - Airbnb 강릉 숙소 파일

작성일: 2026-06-06

상태: feature spec. 이 문서는 강릉 Airbnb 숙소 파일을 제품에 넣고, 체크인 질문에 답하고, 필요한 카드를 남기는 흐름을 잡는다.

## 자료 경계

이번 스펙은 `docs/material-research/2026-06-04-travel-materials-codex/sanitized-airbnb-gangneung-checkin.md`에서 공개 가능한 내용만 가져온다.

제품 입력으로 다룰 파일은 `예약 확인서`와 `호스트 메시지` 두 개다. 공개 스펙에는 Gmail message id, 원본 URL, 예약 코드, 영수증 ID, 결제 정보, 정확한 주소, 지도 좌표, 호스트 식별자, 수신자 정보를 옮기지 않는다.

이 스펙에서 실제 값으로 쓰는 것은 체크인 시작 시간 `15:00`과 체크아웃 시간 `11:00`이다. 늦은 도착 조건은 현재 공개 자료에 구체 값이 없다.

## 짧은 기준

```text
왜 지금: 사용자가 강릉 숙소 파일을 넣고 체크인 질문에 답을 받는 첫 제품 흐름을 닫는다.
사용자 장면: 사용자가 Airbnb 강릉 예약 확인서와 호스트 메시지 파일을 넣고 "체크인은 몇 시부터야? 늦게 도착하면 어떻게 해야 해?"라고 묻는다.
먼저 구현할 흐름: 파일 선택/추가 -> 본문 파싱 -> AI 해석 -> 채팅 답변 -> 인라인 근거.
이번 확인 기준: 체크인 시작 시간은 `근거 있음`, 늦은 도착 조건은 `근거 부족`.
주의할 점: 자료에 없는 늦은 도착 조건을 일반 지식으로 채우지 않는다.
남은 판단: 카드 초안과 대시보드까지 같은 작업에서 이을지, 4-5번 하위 스펙에서 이을지.
```

## 사용자 장면

사용자는 숙소 근처로 이동 중이거나 체크인 전날이다. 사용자는 Airbnb 예약 확인서와 호스트 메시지 파일을 제품에 넣고, 전체 자료함에 아래처럼 묻는다.

```text
체크인은 몇 시부터야? 늦게 도착하면 어떻게 해야 해?
```

사용자는 긴 예약 메일이나 앱 상세 화면을 다시 뒤지지 않고, 답이 어느 자료에 근거하는지와 자료가 말하지 않는 부분을 함께 알고 싶다.

## 목표

사용자는 체크인 시작 시간이 `15:00`임을 근거와 함께 확인한다. 현재 확인된 자료에 늦은 도착 조건의 구체 값이 없으면 AI는 값을 만들지 않고 `근거 부족`으로 멈춘다.

## 규칙

- 사용자가 넣은 파일에서 파싱한 본문으로 답한다.
- 파일 파싱은 1번 하위 스펙의 포함 범위다.
- 파일 본문 없이 답변하지 않는다.
- `근거 있음` 답변은 최소 하나의 인라인 근거를 가진다.
- 늦은 도착 조건은 현재 자료에 구체 값이 없으므로 일반 숙소 지식이나 Airbnb 도움말 지식으로 보충하지 않는다.
- 호스트 메시지의 "판매/숙소 페이지 확인" 안내는 보조 맥락일 뿐, late arrival 조건의 값이 아니다.
- 사용자가 직접 확인해 값을 채워 올리는 흐름은 허용하되, 그 카드는 `직접 확인` 출처로 구분한다.

## 범위

- Gmail 계정 연결, Airbnb 앱 자동 수집, 장기 저장은 이번 흐름에서 요구하지 않는다.
- 모든 파일 형식을 한 번에 지원하지 않는다. 먼저 강릉 숙소 자료 파일 두 개를 안정적으로 읽는다.
- late arrival 조건, 출입 방법, 주차, wifi, house rules를 한 번에 완성하지 않는다.
- 영수증의 환불 정책을 체크인 답변으로 확장하지 않는다. 환불 조건 확인은 별도 질문으로 둔다.
- 실제 Airbnb 원문 본문, Gmail message id, private URL, 예약번호, 정확한 주소, 결제정보를 공개 스펙에 보존하지 않는다.

## 상태 말

| 상황 | 사용자 문구 | 이번 스펙에서의 의미 |
| --- | --- | --- |

## 구현 요소 점검

| 구현 요소 | 이번 장면에서 필요한 이유 | 현재 코드/문서 비교 | 처음 닫을 기준 |
| --- | --- | --- | --- |
| 파일 넣기 / 파싱 | 예약 확인서와 호스트 메시지 파일이 제품의 시작점이어야 한다 | client는 React/Vite이고 `src/client/App.tsx`의 materials state가 비어 있다 | 파일 선택/추가, 본문 읽기, 자료함 표시, AI 해석 입력 연결을 닫는다 |
| AI 해석 | 체크인 시작 시간 후보와 late arrival missing 상태가 필요하다 | `src/server/trip-facts/extractTripFacts.ts`는 현재 late arrival 값을 고정으로 만든다 | 파싱된 본문에서 후보와 근거를 만든다. 본문에 없는 late arrival 값은 만들지 않는다 |
| 정규화 / 상태 말 | 후보를 `근거 있음`, `근거 부족`으로 나눠야 한다 | `src/server/trip-facts/normalizeTripFacts.ts`는 evidence 없는 후보를 `missing`으로 둘 수 있다 | 화면에 보일 상태 말을 shared 상태와 맞춘다 |
| 채팅 답변 | 사용자가 전체 자료함에 묻고 답을 받아야 한다 | client에는 chat workspace shell과 empty draft panel이 있지만 실제 답변 흐름은 아직 닫혀 있지 않다 | 질문, 답변, 상태, 근거 snippet을 한 화면에서 보이게 한다 |
| 인라인 근거 | `근거 있음` 답변은 본문 일부를 보여야 한다 | `EvidenceRef`는 artifact, locator, snippet을 가진다 | 처음은 자료명과 snippet으로 충분하다 |
| 카드 초안 / 대시보드 | 답변이 자동 확정되지 않고 사람이 검토해야 한다 | preview와 PRD는 CardDraft, DashboardCard 경계를 둔다 | 초안, 직접 확인, 확정, 현장 저장 흐름을 순서대로 잇는다 |

주의: `src/server/trip-facts/extractTripFacts.ts`의 현재 고정값 코드는 late arrival 값을 만든다. Airbnb 강릉 자료 기준 제품 흐름에서는 이 값을 근거로 쓰면 안 된다.

## 먼저 구현할 흐름

먼저 구현할 흐름은 `강릉 숙소 파일에서 체크인 답변까지 + 늦은 도착 조건 근거 부족`이다. 하나의 사용자 질문 안에서 `근거 있음`과 `근거 부족`을 같이 확인한다.

이번에 충분히 닫는 면은 `파일 선택/추가 -> 본문 파싱 -> AI 해석 -> 상태 정규화 -> 채팅 답변 -> 인라인 근거`다. 파일 본문 없이 만든 답변은 이 흐름을 대신할 수 없다.

## 확인 기준

1. 사용자가 강릉 숙소 자료 파일 두 개를 넣을 수 있다.
2. 제품은 각 파일의 본문을 읽고 자료함에 보여준다.
3. 사용자가 "체크인은 몇 시부터야?"라고 물으면 답변은 `15:00`을 말하고 `근거 있음`과 인라인 근거를 함께 보여준다.
4. 사용자가 "늦게 도착하면 어떻게 해야 해?"라고 물으면 현재 자료에 구체 조건이 없으므로 값을 만들지 않고 `근거 부족`으로 멈춘다.

## Placeholder 참고

아래 표는 강릉 숙소 자료를 만들 때 원문 세부값을 옮기지 않기 위한 참고다. 기능 요구가 아니라 공개 자료를 만들 때 참고하는 기준이다.

| 원본 필드 유형 | placeholder 표현 | 참고 |
| --- | --- | --- |
| 숙소명/객실명 | `[PROPERTY_TITLE]` | 실제 숙소 식별값을 쓰지 않는다 |
| 호스트명/전화번호 | `[HOST_NAME]`, `[PHONE_NUMBER]` | 개인 또는 사업자 식별값을 쓰지 않는다 |
| 정확한 주소/지도 좌표 | `[EXACT_ADDRESS]`, `[MAP_COORDINATES]` | 실제 방문 위치를 쓰지 않는다 |
| 예약 코드/확인 코드/영수증 ID | `[RESERVATION_CODE]`, `[RECEIPT_ID]` | 예약/영수증 식별자를 쓰지 않는다 |
| private itinerary/message/receipt link | `[PRIVATE_LINK]` | 인증 또는 추적 token이 붙은 URL을 쓰지 않는다 |
| 결제 수단/금액 | `[PAYMENT_METHOD]`, `[AMOUNT]` | 결제 세부값을 쓰지 않는다 |
| 수신자 이름/이메일 | `[GUEST_NAME]`, `[GUEST_EMAIL]` | 개인 수신자 정보를 쓰지 않는다 |
| 출입/도어/락박스 코드 | `[ACCESS_CODE]` | 실제 출입 코드를 쓰지 않는다 |

## 남은 질문

- 첫 강릉 숙소 공개 파일을 `.txt`로 둘지, `.md`로 둘지.
- 4번 카드 초안과 5번 대시보드를 한 구현 턴에 같이 이을지.
- 현재 product model 문서의 일부 타입 설명이 `TripFact.value`를 확장 후보로 말하지만 실제 공유 타입은 이미 `string | null`이다. 이 정리를 별도 작은 문서 정리로 다룰지.
