# Example: 큰 feature spec - 숙소 체크인 확인

이 문서는 `docs/specs/README.md`를 읽을 때 쓰는 calibration sample이다. 큰 feature에서 구현면을 펼치고, slice 후보를 본 뒤, 먼저 고를 slice를 정하는 감각을 보여준다. 현재 작업 queue가 아니며, 아래 slice 후보가 구현 순서나 승인 gate가 되면 안 된다.

## 한눈에 보기

### 사용자 장면

사용자가 호텔 예약 확인서와 호스트 안내 메시지를 자료함에 넣고, 전체 자료함에 묻는다.

```text
체크인은 몇 시부터야? 늦게 도착하면 어떻게 해야 해?
```

### Goal

사용자는 체크인 시작 시간을 근거와 함께 확인하고, 자료에 늦은 도착 조건이 없으면 AI가 값을 만들지 않는 것을 본다. 이후 사용자가 직접 확인한 값만 카드로 올릴 수 있어야 한다.

### Rules

- 자료에 있는 문장으로만 답한다.
- `근거 있음` 답변은 최소 하나의 원문 근거를 가진다.
- 자료에 없는 늦은 도착 조건은 `근거 부족`으로 멈춘다.
- `근거 부족`은 카드 올리기를 막지 않는다. 사용자가 값을 직접 채우면 그 카드는 `직접 확인`으로 표시된다.
- 예약번호, 출입 코드, 연락처 같은 민감 값은 자동 카드가 되지 않는다.

### Non-goals

- 실제 PDF/OCR parser의 품질이나 모든 파일 형식 처리는 먼저 고를 slice의 통과 조건이 아니다. 대신 sanitized text나 seed artifact로 자료함에서 들어온 정보가 채팅 답변까지 닿는 경로는 살린다.
- 숙소 체크인 전체 필드, 모든 voucher 형식, full card editing UX를 한 번에 완성하지 않는다. 단 카드 초안과 대시보드 출처 경계는 이번 product thread에서 관찰한다.
- `conflict`, 세밀한 masking, eval metric threshold는 먼저 고를 slice의 통과 조건이 아니다. product behavior가 생긴 뒤 별도 slice나 eval 관찰로 연다.
- 별도 work-map이나 execution-map 파일을 만들지 않는다.

### 상태 언어

| 상황 | 사용자 문구 | 이 예시에서의 의미 |
| --- | --- | --- |
| 자료에서 체크인 시간이 확인됨 | 근거 있음 | 답변과 후보가 원문 근거를 가진다 |
| 늦은 도착 조건이 자료에 없음 | 근거 부족 | 값을 만들지 않고 직접 확인 여지를 남긴다 |
| 사용자가 직접 값을 입력해 카드로 올림 | 직접 확인 | AI 근거 카드와 구분되는 대시보드 카드가 된다 |
| 출입 코드처럼 민감한 값이 보임 | 확인 필요 | 자동 저장하지 않고 원문 확인 대상으로 둔다 |

### 구현면 펼치기

구현면은 작업 목록이 아니라, 이번 사용자 장면을 화면까지 통과시키려면 어떤 큰 면을 지나야 하는지 보는 지도다.

| 구현면 | 이번 장면에서 필요한 이유 | 현재 코드 비교 | 이번에 열어둘 경계 |
| --- | --- | --- | --- |
| 자료 입력 / ingest | 예약 확인서와 호스트 안내 메시지가 자료함에 있어야 질문이 시작된다 | 자료함 UI는 있지만 실제 입력 경로는 아직 얇다 | sanitized seed artifact로 시작하고 실제 PDF/OCR/Gmail ingestion은 뒤로 둔다 |
| AI 후보 생성 / adapter | 체크인 시작 시간과 늦은 도착 조건 후보가 필요하다 | baseline adapter가 deterministic 후보를 만들 수 있다 | real LLM provider는 뒤로 둔다 |
| 정규화 / 상태 언어 | 후보를 `근거 있음`, `근거 부족`, `확인 필요`로 나눠야 한다 | `EvidenceState`, `EvidenceRef`, `TripFact` 계약이 있다 | 세밀한 missing taxonomy는 뒤로 둔다 |
| 채팅 답변 | 사용자가 전체 자료함에 질문하고 답을 받아야 한다 | 채팅 화면은 있지만 상태/evidence 렌더링은 이어야 한다 | rich chat UX 대신 상태 문구와 답변 block부터 둔다 |
| 인라인 근거 | `근거 있음` 답변은 snippet을 보여야 한다 | evidence 계약은 있지만 화면 경로를 이어야 한다 | 이미지 영역 근거 표시는 뒤로 둔다 |
| 카드 초안 | 답변을 카드로 올리기 전 검토 경계가 필요하다 | 카드 초안 영역은 있지만 empty state 중심이다 | full card editing UX는 뒤로 둔다 |
| 직접 확인 / 대시보드 출처 | 사용자가 직접 채운 값은 AI 근거 카드와 구분되어야 한다 | 출처 언어는 정리되어 있지만 카드 출처 경로는 이어야 한다 | 실제 연락 workflow와 persistence는 뒤로 둔다 |
| 민감정보 guard | 예약번호/출입코드가 자동 카드가 되면 안 된다 | sensitive 후보 표현은 가능하다 | 세밀한 masking taxonomy는 뒤로 둔다 |

### Slice 후보

Slice 후보는 위 구현면을 얇게 관통하는 product path다. 후보가 적혀 있어도 모두 같은 구현 턴에 닫는다는 뜻은 아니다.

| 후보 | 관통하는 구현면 | 실제로 닫는 제품 동작 | stub으로 둘 것 | 선택 판단 |
| --- | --- | --- | --- | --- |
| 체크인 시작 시간 답변 | 자료 입력 / AI 후보 / 정규화 / 채팅 / 인라인 근거 | 사용자가 질문하면 `15:00`이 `근거 있음`과 snippet으로 보인다 | real ingestion, real LLM, 카드/대시보드 | 첫 product proof로 가장 좁고 깊게 닫기 좋은 후보 |
| 늦은 도착 조건 근거 부족 | 자료 입력 / AI 후보 / 정규화 / 채팅 | 자료에 값이 없으면 값을 만들지 않고 `근거 부족`으로 멈춘다 | 직접 확인 카드, missing taxonomy 전체 | 체크인 답변과 같이 묶거나 다음 slice로 분리할 수 있는 후보 |
| 답변에서 카드 초안으로 | 채팅 / 카드 초안 / 민감정보 guard | 근거 있는 답변을 카드 초안으로 올리기 전 검토한다 | dashboard persistence, 카드 편집 전체 UX | 답변이 먼저 보인 뒤 열면 자연스러운 후보 |
| 근거 부족에서 직접 확인 카드로 | 채팅 / 직접 확인 입력 / 대시보드 출처 | 사용자가 직접 채운 값이 `직접 확인`으로 보인다 | 실제 연락/원문 확인 workflow | `근거 부족` 답변이 보인 뒤 이어야 할 후보 |
| 민감정보 자동 카드 제외 | AI 후보 / 정규화 / 카드 초안 / guard | 예약번호/출입코드가 자동 카드 후보가 되지 않는다 | 세밀한 masking taxonomy | 카드 경로가 열릴 때 같이 확인할 보호 후보 |

### 먼저 고를 slice

먼저 고르기 좋은 slice는 `체크인 시작 시간 답변`이다. sanitized seed와 deterministic adapter로 `자료 -> AI/adapter -> 상태 -> 채팅 답변 -> 인라인 근거`를 화면까지 잇는다.

이 선택은 나머지 후보를 버린다는 뜻이 아니다. `늦은 도착 조건 근거 부족`, `직접 확인 카드`, `민감정보 자동 카드 제외`는 같은 product scene의 후보로 남기고, 다음 작업 brief에서 같이 닫을지 별도 slice로 뺄지 다시 고른다.

아래 AC 3개는 feature proof에서 관찰할 제품 동작이다. 특정 구현 턴의 작업 목록으로 고정하지 않는다.

### 이번 AC

1. 자료에 `체크인 15:00` 문장이 있으면 채팅 답변에 `근거 있음`과 인라인 근거가 함께 보인다.
2. 자료에 늦은 도착 조건이 없으면 답변은 값을 만들지 않고 `근거 부족`으로 멈춘다.
3. 사용자가 늦은 도착 조건을 직접 입력해 카드로 올리면 대시보드에서는 `직접 확인` 출처로 보인다.

### 확인 방법

- sanitized text나 seed artifact에 체크인 시작 시간만 두고 늦은 도착 조건은 넣지 않는다.
- 화면에서 체크인 시간 답변의 근거 snippet이 실제 값과 맞는지 확인한다.
- 늦은 도착 답변이 확정형 문장이나 카드로 자동 승격되지 않는지 확인한다.
- 사용자가 값을 직접 채운 뒤 대시보드 카드의 출처 문구가 `직접 확인`인지 확인한다.

## 상세 기준

### Flow

```text
자료함에 예약 확인서/호스트 안내 추가
-> 전체 자료함에 체크인 질문
-> ChatAnswer: 체크인 시작 시간은 근거 있음
-> ChatAnswer: 늦은 도착 조건은 근거 부족
-> 사용자가 늦은 도착 조건을 직접 채워 CardDraft 생성
-> DashboardCard: 직접 확인
```

### Tests / scenario(사례)

| scenario | 기대 동작 | 실패 냄새 |
| --- | --- | --- |
| 체크인 시작 시간이 자료에 있음 | `15:00` + 원문 근거 + `근거 있음` | confidence 수치만 보이거나 근거 snippet이 값과 다름 |
| 늦은 도착 조건이 자료에 없음 | `근거 부족`, 값을 만들지 않음 | "일반적으로 호텔은..." 같은 추정 답변 |
| 사용자가 직접 값을 입력함 | 대시보드 카드가 `직접 확인` | AI 근거 카드처럼 `근거 있음`으로 표시 |

### 보류 질문

- 민감 정보 snippet masking은 `[masked]` 수준으로 충분한가?
