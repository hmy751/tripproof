# TripProof PRD

## 1. Product Concept

TripProof는 이번 여행 자료를 넣으면, 사용자가 행동 전에 "이 정보를 믿고 움직여도 되는가"를 상황별 신뢰 상태로 가려주는 여행 자료 검증 도구다.

핵심 명사는 채팅이나 카드가 아니라 **검증된 신뢰 상태**다.

TripProof는 일반 여행 일정 앱, 문서 보관함, PDF 챗봇, 월렛 앱과 다르게 포지셔닝한다.

- 일정 앱은 예약을 시간표로 정리한다.
- 문서 보관함은 원본 자료를 모아둔다.
- PDF 챗봇은 자료에 대한 답변을 생성한다.
- 월렛 앱은 현장에서 제시할 패스를 저장한다.
- TripProof는 자료 속 조건을 근거와 상태로 검증하고, 사용자가 저장한 정보만 현장 카드로 굳힌다.

TripProof의 핵심 출력은 자료 요약이나 채팅 로그가 아니라 다음과 같은 상태 보드다.

```text
상황별 신뢰 상태
- 숙소 체크인
- 투어 출발
- 렌터카 반납
- 결제 확인

각 상황 안의 정보
- 값
- 신뢰 상태
- 원문 근거
- 다음 행동
- 현장 저장 여부
```

## 2. Core Promise

```text
이번 여행 자료를 넣으면,
TripProof가 행동 전에 확인할 상황을 찾고,
각 정보가 확인됨/검토 필요/근거 부족/자료 충돌 중 어디에 있는지 보여주며,
근거와 현장 재사용 가치가 있는 정보만 사용자가 현장 카드로 저장한다.
```

제품이 지켜야 할 가장 중요한 약속은 다음 네 가지다.

1. 자료를 행동 상황으로 바꾼다.
2. 정보마다 신뢰 상태와 근거를 붙인다.
3. 질문에는 답변하되, 답변을 자동으로 카드화하지 않는다.
4. 저장한 정보만 현장에서 쓰게 한다.

## 3. Target User

TripProof의 주요 사용자는 직접 여행 자료를 관리하고, 현장에서 중요한 조건을 스스로 확인해야 하는 개인 여행자다.

이 사용자는 다음과 같은 상황에 자주 놓인다.

- 예약 화면, PDF, 메시지, 영수증이 여러 앱에 흩어져 있다.
- 이동 중이라 긴 문서를 다시 읽기 어렵다.
- 체크인, 투어 출발, 렌터카 반납, 추가 결제 확인처럼 실수 비용이 큰 순간이 있다.
- AI가 답을 줘도 근거가 없으면 믿기 어렵다.
- 민감 정보가 자동 저장되거나 잘못 노출되는 것은 부담스럽다.

## 4. Product Model

TripProof는 다음 객체를 중심으로 동작한다.

### Trip

사용자가 확인하려는 하나의 여행 묶음이다.

예:

```text
오사카 4박 5일
```

### Artifact

사용자가 넣은 원본 자료다. Artifact는 최종 목적지가 아니라 근거 레이어다.

예:

```text
호텔 예약 화면
호스트 체크인 안내
후지산 투어 바우처
렌터카 예약 약관
호텔 도시세 영수증
```

Artifact는 여러 Situation의 근거로 연결될 수 있다.

### Situation

사용자가 실제로 행동해야 하는 여행 순간이다. TripProof의 주 화면 단위다.

예:

```text
숙소 체크인
투어 출발
렌터카 반납
결제 확인
```

### Verification Item

Situation 안에서 신뢰 상태를 확인해야 하는 정보 조각이다.

예:

```text
체크인 시간
늦은 도착 조건
모바일 바우처 가능 여부
집합 시간
연료 정책
도시세
```

Verification Item은 두 경로로 생길 수 있다.

```text
AI가 자료에서 먼저 찾은 행동 전 확인 항목
채팅 답변 중 현장에서 다시 볼 가치가 있어 사용자가 승격한 항목
```

중요한 점은 채팅 답변이 항상 Verification Item이 되는 것은 아니라는 점이다.

### Chat Answer

사용자가 자료에 질문했을 때 받는 답변이다.

Chat Answer는 항상 다음 구조를 가져야 한다.

```text
답변
상태
근거
연결된 상황
다음 액션
```

하지만 다음 액션은 질문의 성격에 따라 달라진다.

```text
값과 근거가 있고 현장에서 다시 볼 필요가 있음:
- 현장 정보로 저장
- 관련 상황에 확인 항목으로 올리기

이해만 하면 되는 질문:
- 답변으로 남기기
- 근거 보기

근거가 부족한 질문:
- 자료 추가
- 근거 부족으로 남기기

자료가 충돌하는 질문:
- 충돌 항목으로 표시
- 원문 비교

민감 정보:
- 원문에서 직접 확인
- 자동 저장하지 않기
```

### Evidence

Verification Item 또는 Chat Answer가 나온 원문 근거다.

Evidence는 최소한 다음을 가져야 한다.

```text
자료 이름
자료 종류
찾은 위치
짧은 원문
```

### Field Card

사용자가 근거를 보고 저장한 정보만 모인 현장용 카드다.

Field Card는 검증 축의 도착점이다. 채팅과 경쟁하는 화면이 아니라, 검증을 통과했거나 사용자가 현장에서 다시 보기로 결정한 정보만 담는 결과물이다.

검토하지 않은 값, 민감 정보, 근거 부족 항목은 Field Card에 자동으로 들어가면 안 된다.

## 5. Information States

TripProof는 정보의 신뢰 상태를 명확히 분리한다.

| 상태 | 사용자 문구 | 의미 |
| --- | --- | --- |
| supported | 확인됨 | 자료 근거로 확인된 정보 |
| needs_review | 확인 필요 | 민감하거나 애매해서 직접 확인이 필요한 정보 |
| missing | 근거 부족 | 현재 자료에서 근거를 찾지 못한 정보 |
| conflict | 자료 충돌 | 서로 다른 자료가 다른 값을 말하는 정보 |
| saved | 저장됨 | 사용자가 현장에서 보기로 저장한 정보 |

상태는 내부 진단용이 아니라 사용자 행동을 막거나 허용하는 UX 언어다.

나쁜 표현:

```text
신뢰도 0.84
근거 선명 88%
AI confidence medium
```

좋은 표현:

```text
확인됨
확인 필요
근거 부족
자료 충돌
저장됨
민감 정보
```

## 6. Core User Scenario

### 1. 사용자가 여행 자료를 넣는다

사용자는 오사카 여행을 앞두고 여러 자료를 TripProof에 넣는다.

```text
호텔 예약 화면
호스트 체크인 안내 메시지
투어 바우처 PDF
렌터카 약관 PDF
호텔 도시세 영수증
```

이 입력은 "아무 자료나 넣기"가 아니다. 사용자는 이번 여행과 관련된 원본 자료 묶음을 넣는다.

중심 문구:

```text
이번 여행 자료를 넣으세요.
TripProof가 체크인, 투어, 반납, 결제 확인처럼 행동 전에 확인할 상황으로 나눕니다.
```

### 2. AI가 상황별 신뢰 상태를 만든다

TripProof는 자료를 읽고 사용자가 행동해야 할 상황을 먼저 제안한다.

예:

```text
숙소 체크인
- 확인됨: 체크인 시간, 늦은 도착 조건, 도시세
- 확인 필요: 예약번호, 출입 코드

투어 출발
- 확인됨: 모바일 바우처, 집합 시간
- 확인 필요: 집합 장소
- 근거 부족: 지각 환불 규정

렌터카 반납
- 확인됨: 연료 정책
- 확인 필요: 보증금 조건
- 근거 부족: 반납 위치
```

이 단계는 카드 목록을 많이 만드는 것이 아니라, 사용자가 지금 무엇을 믿고 무엇을 확인해야 하는지 상태로 나누는 단계다.

### 3. 사용자가 자료에 질문한다

상태 보드만으로 부족하면 사용자는 자료에 질문한다.

예:

```text
호텔에 늦게 도착하면 어떻게 돼?
투어 바우처 프린트해야 해?
렌터카 반납 전에 기름 채워야 해?
이 영수증은 왜 낸 거야?
투어 늦으면 환불돼?
```

채팅은 별도 챗봇 기능도, 카드와 5:5로 경쟁하는 화면도 아니다.

채팅의 역할은 **상황별 신뢰 상태 뷰에 던지는 contextual probe**다. AI가 먼저 덮지 못한 불안을 사용자가 직접 찌르는 입력 방식이다.

### 4. 질문에는 답변하되, 결말은 분기된다

모든 질문은 그 자리에서 답변을 받는다.

예:

```text
질문:
호텔에 늦게 도착하면 어떻게 돼?

답변:
22:00 이후 도착하면 숙소에 연락해야 합니다.

상태:
확인됨

근거:
- 호텔 예약 화면 · 늦은 도착 안내
- 호스트 체크인 안내 · 셀프 체크인 안내

연결된 상황:
숙소 체크인

액션:
- 현장 정보로 저장
- 근거 보기
```

근거가 없는 질문은 답을 만들어내지 않는다.

예:

```text
질문:
투어 늦으면 환불돼?

답변:
현재 자료에서는 지각 시 환불 규정을 찾지 못했습니다.

상태:
근거 부족

연결된 상황:
투어 출발

액션:
- 관련 자료 추가
- 근거 부족으로 남기기
```

이해만 하면 되는 질문은 카드로 승격하지 않는다.

예:

```text
질문:
이 영수증은 왜 낸 거야?

답변:
호텔 도시세 JPY 1,200 결제 영수증으로 보입니다.

상태:
확인됨

액션:
- 근거 보기
- 답변으로 남기기
```

### 5. 사용자가 근거를 보고 저장한다

사용자는 AI가 만든 상태를 그대로 믿지 않는다. 값, 상태, 원문 근거를 보고 현장에서 써도 되는 정보만 저장한다.

예:

```text
체크인 시간
값: 15:00
상태: 확인됨
근거 자료: 호텔 예약 화면
찾은 위치: 예약 요약 영역
원문: 체크인 15:00. 22:00 이후 도착 시 숙소 연락 필요.
```

사용자가 저장하면 이 정보는 Field Card로 승격된다.

민감 정보와 근거 부족 항목은 자동 저장하지 않는다.

### 6. 현장에서는 저장한 정보만 본다

현장 카드에는 사용자가 저장한 정보만 표시된다.

예:

```text
숙소 체크인 카드
- 체크인 시간: 15:00
- 늦은 도착: 22:00 이후 숙소 연락 필요
- 도시세: JPY 1,200
```

다음 정보는 현장 카드에서 분리된다.

```text
예약번호: 민감 정보, 직접 확인
출입 코드: 민감 정보, 직접 확인
정확한 입구: 근거 부족
지각 환불 규정: 근거 부족
```

## 7. UX Structure

TripProof의 핵심 화면은 네 개의 동등한 탭이 아니라 **상황별 신뢰 상태 뷰**다.

```text
메인 척추:
상황별 신뢰 상태 보드

보조 레이어:
- 자료 입력/자료함
- 질문 probe
- 근거 드릴다운
- 현장 카드
```

### 자료 입력/자료함

역할:

- 원본 자료를 받는다.
- 자료가 어떤 상황의 근거로 쓰이는지 보여준다.
- 사용자가 새 자료를 추가할 수 있게 한다.

자료 화면의 중심 문구:

```text
이번 여행 자료를 넣으세요.
자료는 원본으로 남고, TripProof가 행동 전에 확인할 상황으로 연결합니다.
```

### 상황별 신뢰 상태 보드

역할:

- TripProof의 중심 화면이다.
- AI가 먼저 찾은 상황을 보여준다.
- 각 상황 안에서 확인됨, 확인 필요, 근거 부족, 자료 충돌, 저장됨을 분리한다.
- 사용자가 무엇을 믿고, 무엇을 확인하고, 무엇을 저장했는지 한 화면에서 보게 한다.

구성:

```text
상단 요약
- 자료 5개 분석됨
- 확인됨 7개
- 확인 필요 3개
- 근거 부족 2개
- 저장됨 4개

상황별 상태
- 숙소 체크인
- 투어 출발
- 렌터카 반납
- 결제 확인

각 상황의 항목
- 값
- 상태
- 근거
- 다음 행동
```

### 질문 Probe

역할:

- 사용자가 AI가 먼저 덮지 못한 불안을 묻게 한다.
- 답변은 관련 상황에 연결된다.
- 답변이 항상 카드가 되는 것은 아니다.

구성:

```text
질문 입력
추천 질문
답변
상태
근거
다음 액션
```

### 근거 드릴다운

역할:

- 정보 하나를 열어 값과 근거를 검토한다.
- 저장, 원문 확인, 자료 추가, 충돌 비교 같은 액션을 제공한다.

구성:

```text
값
상태
근거 자료
찾은 위치
원문
액션
```

### 현장 카드

역할:

- 저장된 정보만 상황별로 보여준다.
- 여행 중 빠르게 볼 수 있는 화면이다.
- 저장되지 않은 정보가 행동 정보처럼 보이지 않게 한다.

구성:

```text
숙소 체크인
- 체크인 시간
- 늦은 도착 조건
- 저장된 결제 참고

투어 출발
- 모바일 바우처
- 집합 시간
```

## 8. Chat And Card Integration

카드와 채팅은 서로 경쟁하는 기능이 아니다.

역할은 다음처럼 나뉜다.

```text
상황별 신뢰 상태 보드:
제품의 중심 화면

채팅:
상태 보드에 던지는 질문 probe

답변:
상태와 근거를 가진 대화 결과

현장 카드:
저장할 가치가 있다고 판단한 정보의 결과물
```

채팅이 제품을 흐리게 만들지 않으려면 다음 규칙이 필요하다.

1. 질문에는 항상 답변한다.
2. 답변은 근거 없이 확정형으로 끝나면 안 된다.
3. 답변은 상태를 가져야 한다.
4. 답변은 가능한 한 관련 Situation에 연결한다.
5. 답변은 자동으로 카드가 되지 않는다.
6. 값, 근거, 현장 재사용 가치가 있을 때만 저장 액션을 제공한다.
7. 근거가 없으면 `근거 부족` 상태로 남기고 관련 자료 추가를 유도한다.
8. 충돌이 있으면 `자료 충돌` 상태로 남기고 원문 비교를 제공한다.
9. 민감 정보는 자동 저장하지 않고 원문 확인으로 보낸다.

## 9. MVP Scope

초기 MVP는 전체 여행 자동 관리가 아니라, 한 여행 자료 묶음에서 다음 흐름을 보여주는 데 집중한다.

포함:

- 이번 여행 자료 추가
- 자료함
- 상황별 신뢰 상태 보드
- AI가 먼저 찾은 행동 전 확인 항목
- 자료에 질문하기
- 답변에 상태와 근거 붙이기
- 답변의 조건부 저장/승격
- 항목별 근거 드릴다운
- 저장한 정보만 현장 카드로 표시
- 민감 정보, 근거 부족, 충돌 항목 분리

우선 지원할 상황:

```text
숙소 체크인
투어 출발
렌터카 반납
결제 확인
```

우선 지원할 자료:

```text
예약 화면 이미지
호스트 메시지 캡처
바우처 PDF
약관 PDF
영수증 이미지
```

제외:

- 완전한 여행 일정 생성
- 예약, 결제, 환불 처리
- 항공편 실시간 상태 조회
- 범용 문서 금고
- 근거 없는 일반 여행 상담
- 사용자 확인 없는 자동 저장
- 모든 채팅 답변의 자동 카드화

## 10. Success Criteria

TripProof의 UX가 성공하려면 사용자가 다음을 이해할 수 있어야 한다.

1. 자료를 넣으면 상황별 신뢰 상태가 만들어진다.
2. AI가 먼저 확인 항목을 찾지만, 사용자가 추가 질문할 수도 있다.
3. 질문하면 답변은 받되, 모든 답변이 카드가 되는 것은 아니다.
4. 모든 답에는 원문 근거와 상태가 붙는다.
5. 값, 근거, 현장 재사용 가치가 있는 정보만 저장할 수 있다.
6. 저장한 정보만 현장 카드에 들어간다.
7. 근거 부족, 충돌, 민감 정보는 행동 정보처럼 보이지 않는다.

실패 신호:

- 사용자가 TripProof를 그냥 PDF 챗봇으로 이해한다.
- 사용자가 TripProof를 그냥 자료 보관함으로 이해한다.
- 채팅과 카드가 동급 기능처럼 싸운다.
- 질문 답변이 모두 자동 카드화된다.
- 근거 없는 답변이 확인된 정보처럼 보인다.
- 검토하지 않은 정보가 현장 카드에 들어간다.
- 민감 정보가 자동 저장된다.
