# TripProof Preview PRD

## 1. Product Concept

TripProof는 여행 자료함에 넣은 PDF, 캡처, 메시지, 영수증을 기반으로 AI와 질문하고, 사용자가 근거 있는 답변을 검토한 뒤 카드로 올려 최종 대시보드와 현장 카드로 정리하는 여행 자료 확인 도구다.

핵심 명사는 하나가 아니다. 이 preview의 핵심은 네 객체가 연결되는 흐름이다.

```text
자료함
→ 자료 기반 채팅
→ 카드 초안
→ 최종 대시보드
→ 현장 카드
```

TripProof는 일반 PDF 챗봇처럼 답변 로그만 쌓지 않는다. 그렇다고 AI가 자료를 읽자마자 최종 대시보드를 자동 완성하지도 않는다.

TripProof의 차별점은 다음이다.

- 사용자는 자료함에 여행 원본을 넣는다.
- AI는 자료를 기반으로 답변하고, 먼저 확인해볼 후보도 제안한다.
- 사용자는 답변과 근거를 보고 괜찮다고 판단한 정보만 카드로 올린다.
- 최종 대시보드는 사람이 올린 카드만 담는다.
- 현장 카드는 최종 대시보드에서 다시 보기로 저장한 정보만 담는다.

## 2. Core Promise

```text
이번 여행 자료를 넣으면,
TripProof가 자료에 대해 답하고 확인 후보를 제안하며,
사용자가 근거를 보고 괜찮은 답변만 카드로 올리고,
현장에서 다시 볼 정보만 저장하게 해준다.
```

제품이 지켜야 할 약속은 다음이다.

1. 채팅은 실제로 개발되는 핵심 인터랙션이다.
2. 채팅 답변은 항상 자료 근거와 상태를 가져야 한다.
3. 채팅 답변은 자동으로 카드가 되지 않는다.
4. 사용자가 답변을 보고 선택해야 카드 초안이 된다.
5. 사용자가 확정해야 최종 대시보드에 올라간다.
6. 현장 카드는 최종 대시보드에서 저장한 정보만 담는다.

## 3. Target User

TripProof의 사용자는 여행 전에 여러 앱과 파일에 흩어진 자료를 확인해야 하는 개인 여행자다.

이 사용자는 다음 상황에 놓인다.

- 호텔 예약 화면, 바우처 PDF, 호스트 메시지, 영수증이 흩어져 있다.
- 여행 중 또는 출발 직전이라 긴 문서를 다시 읽기 어렵다.
- AI에게 물어보고 싶지만 근거 없는 답은 믿기 어렵다.
- 답변이 괜찮으면 나중에 다시 볼 수 있게 카드로 남기고 싶다.
- 민감 정보나 근거 부족 답변이 자동 저장되는 것은 부담스럽다.

## 4. Product Model

### Trip

사용자가 확인하려는 하나의 여행 묶음이다.

예:

```text
오사카 4박 5일
```

### Artifact

사용자가 자료함에 넣은 원본 자료다.

예:

```text
호텔 예약 화면
호스트 체크인 안내
후지산 투어 바우처
렌터카 예약 약관
호텔 도시세 영수증
```

Artifact는 채팅 답변과 카드의 근거가 된다.

### Library Chat

사용자가 자료함에 대해 묻는 채팅이다.

예:

```text
호텔에 늦게 도착하면 어떻게 돼?
투어 바우처 프린트해야 해?
이 영수증은 왜 낸 거야?
투어 늦으면 환불돼?
```

Library Chat은 단순 보조 기능이 아니다. 자료를 탐색하고 카드 후보를 발견하는 주 인터랙션이다.

### Chat Answer

자료 기반 채팅의 답변이다.

Chat Answer는 최소한 다음을 가져야 한다.

```text
답변
상태
근거 자료
원문 위치
연결 가능한 상황
다음 액션
```

답변의 결말은 질문 성격에 따라 갈린다.

```text
값과 근거가 있고 다시 볼 정보:
- 카드로 올리기
- 근거 상세 보기

이해만 하면 되는 답변:
- 답변으로 남김
- 근거 상세 보기

근거가 부족한 답변:
- 관련 자료 첨부
- 근거 부족으로 남김

민감 정보:
- 원문에서 직접 확인
- 자동 카드화 금지
```

### AI Candidate

자료를 넣었을 때 AI가 먼저 제안하는 확인 후보다.

AI Candidate는 최종 대시보드 카드가 아니다. 사용자가 검토하기 전의 제안이다.

예:

```text
숙소 체크인 · 늦은 도착
투어 출발 · 바우처 제시
렌터카 반납 · 연료 정책
결제 확인 · 도시세
```

### Card Draft

Chat Answer 또는 AI Candidate를 사용자가 선택했을 때 생기는 중간 확인 단계다.

Card Draft는 다음을 보여준다.

```text
상황
카드 이름
값
상태
근거
원문 위치
```

사용자가 여기서 확정해야 최종 대시보드에 올라간다.

### Dashboard Card

사용자가 카드 초안을 확정해서 최종 대시보드에 올린 카드다.

Dashboard Card는 TripProof의 작업판이다.

```text
숙소 체크인
- 체크인 시간: 15:00
- 늦은 도착: 22:00 이후 숙소 연락 필요

투어 출발
- 바우처 제시: 모바일 바우처 가능
```

### Evidence Detail

답변, 후보, 카드가 가리키는 원문 근거다.

Evidence Detail은 다음을 보여준다.

```text
자료 이름
자료 종류
찾은 위치
짧은 원문
연결된 답변 또는 카드
```

### Field Card

최종 대시보드에서 사용자가 현장에서 다시 보기로 저장한 정보다.

Field Card에는 검토하지 않은 답변, 근거 부족 답변, 민감 정보가 자동으로 들어가면 안 된다.

## 5. Information States

| 상태 | 사용자 문구 | 의미 |
| --- | --- | --- |
| supported | 근거 있음 | 현재 자료 근거로 답할 수 있음 |
| needs_review | 확인 필요 | 민감하거나 애매해서 원문 확인이 필요함 |
| missing | 근거 부족 | 현재 자료로는 답할 수 없음 |
| conflict | 자료 충돌 | 서로 다른 자료가 다른 값을 말함 |
| saved | 현장 저장 | 사용자가 현장에서 보기로 저장함 |

나쁜 표현:

```text
AI confidence 0.84
신뢰도 medium
```

좋은 표현:

```text
근거 있음
확인 필요
근거 부족
자료 충돌
현장 저장
```

## 6. Core User Scenario

### 1. 사용자가 자료를 첨부한다

중심 문구:

```text
이번 여행 자료를 첨부하세요.
```

자료함에는 원본이 남는다.

```text
호텔 예약 화면
호스트 메시지
투어 바우처 PDF
렌터카 약관 PDF
도시세 영수증
```

### 2. AI가 추천 후보를 만든다

자료를 넣으면 AI는 먼저 확인해볼 후보를 제안한다.

예:

```text
숙소 체크인 · 늦은 도착
투어 출발 · 바우처 제시
렌터카 반납 · 연료 정책
결제 확인 · 도시세
```

이 후보는 최종 대시보드가 아니다. 사용자가 선택해야 카드 초안이 된다.

### 3. 사용자가 자료함에 묻는다

사용자는 자료함 기반 채팅으로 직접 묻는다.

예:

```text
호텔에 늦게 도착하면 어떻게 돼?
```

답변:

```text
22:00 이후 도착하면 숙소에 연락해야 합니다.
```

근거:

```text
호텔 예약 화면 · 늦은 도착 안내
호스트 메시지 · 셀프 체크인 안내
```

액션:

```text
카드로 올리기
근거 상세
답변으로 남김
```

### 4. 사용자가 답변을 카드 초안으로 올린다

사용자가 `카드로 올리기`를 누르면 카드 초안이 생긴다.

```text
상황: 숙소 체크인
카드 이름: 늦은 도착
값: 22:00 이후 숙소 연락 필요
상태: 근거 있음
근거: 호텔 예약 화면 + 호스트 메시지
```

사용자가 확정하면 최종 대시보드에 올라간다.

### 5. 최종 대시보드에서 관리한다

최종 대시보드는 AI가 자동으로 채운 보드가 아니라, 사용자가 올린 카드의 작업판이다.

여기서 사용자는 근거를 다시 보고, 현장에서 다시 볼 정보만 저장한다.

### 6. 현장 카드로 저장한다

현장 카드에는 최종 대시보드에서 저장한 정보만 들어간다.

예:

```text
숙소 체크인
- 체크인 시간: 15:00
- 늦은 도착: 22:00 이후 숙소 연락 필요
```

## 7. UX Structure

TripProof preview는 다음 화면 위계를 가진다.

```text
왼쪽:
- 여행 요약
- 자료함
- AI 추천 후보

중앙:
- 현재 흐름
- 자료함 채팅
- 카드 초안
- 최종 대시보드

오른쪽:
- 근거 상세
- 현장 카드
```

### 자료함

역할:

- 원본 자료를 보존한다.
- 채팅과 근거의 출처를 보여준다.
- 새 자료를 첨부하게 한다.

### AI 추천 후보

역할:

- AI가 먼저 확인할 만한 정보를 제안한다.
- 사용자가 후보를 카드 초안으로 보낼 수 있게 한다.
- 확정 대시보드처럼 보이면 안 된다.

### 자료함 채팅

역할:

- 사용자가 자료에 직접 묻는다.
- 답변에 상태와 근거를 붙인다.
- 답변을 카드 초안으로 올리는 입구가 된다.

### 카드 초안

역할:

- 답변이나 후보가 카드가 되기 전, 사람이 내용을 확인하는 단계다.
- 상황, 제목, 값, 근거를 보여준다.
- 확정 액션을 제공한다.

### 최종 대시보드

역할:

- 사용자가 확정한 카드만 모인다.
- 여행 전 확인해야 할 정보의 작업판이다.
- 여기서 현장 카드로 저장한다.

### 근거 상세

역할:

- 답변, 후보, 대시보드 카드가 가리키는 원문을 보여준다.
- 사용자가 카드로 올리거나 저장하기 전에 확인할 수 있게 한다.

### 현장 카드

역할:

- 최종 대시보드에서 저장한 정보만 보여준다.
- 현장에서 빠르게 다시 볼 정보의 도착점이다.

## 8. Chat And Card Integration

채팅은 낮은 보조 기능이 아니다. 다만 채팅 로그 자체가 최종 산출물도 아니다.

역할은 다음처럼 나뉜다.

```text
자료함:
원본과 질문 대상

채팅:
자료를 탐색하고 답변을 받는 핵심 인터랙션

답변:
근거와 상태가 붙은 카드 후보

카드 초안:
사용자가 답변을 구조화해서 검토하는 중간 단계

최종 대시보드:
사용자가 올린 카드의 작업판

현장 카드:
사용자가 다시 보기로 저장한 결과물
```

채팅이 제품을 흐리게 만들지 않으려면 다음 규칙이 필요하다.

1. 질문에는 답변한다.
2. 답변은 근거 없이 확정형으로 끝나면 안 된다.
3. 답변은 상태를 가져야 한다.
4. 답변은 가능한 경우 연결 상황을 가져야 한다.
5. 답변은 자동으로 카드가 되지 않는다.
6. 사용자가 `카드로 올리기`를 눌러야 카드 초안이 된다.
7. 사용자가 카드 초안을 확정해야 최종 대시보드에 올라간다.
8. 근거 부족 답변은 카드로 올리지 않고 자료 첨부를 유도한다.
9. 민감 정보는 자동 저장하지 않고 원문 확인으로 보낸다.

## 9. MVP Scope

포함:

- 이번 여행 자료 첨부
- 자료함
- AI 추천 후보
- 자료함 기반 채팅
- 답변의 상태와 근거 표시
- 답변에서 카드 초안 만들기
- 후보에서 카드 초안 만들기
- 최종 대시보드에 카드 올리기
- 근거 상세 보기
- 최종 대시보드에서 현장 카드 저장

제외:

- 예약, 결제, 환불 처리
- 항공편 실시간 상태 조회
- 범용 문서 금고
- 근거 없는 일반 여행 상담
- 사용자 확인 없는 자동 카드 생성
- 사용자 확인 없는 현장 카드 저장

## 10. Success Criteria

TripProof의 UX가 성공하려면 사용자가 다음을 이해할 수 있어야 한다.

1. 자료함에 자료를 넣고 그 자료에 물어본다.
2. 채팅 답변에는 근거와 상태가 붙는다.
3. 답변을 보고 괜찮다고 판단하면 카드로 올릴 수 있다.
4. 추천 후보도 바로 확정이 아니라 카드 초안으로 간다.
5. 최종 대시보드에는 사람이 올린 카드만 들어간다.
6. 현장 카드에는 최종 대시보드에서 저장한 정보만 들어간다.
7. 근거 부족, 민감 정보, 충돌 정보는 자동 저장되지 않는다.

실패 신호:

- 채팅이 보조 기능처럼 보인다.
- TripProof가 그냥 PDF 챗봇처럼 보인다.
- AI가 대시보드를 자동 완성한 것처럼 보인다.
- 답변이 곧바로 최종 카드가 되는 것처럼 보인다.
- 추천 후보와 최종 대시보드가 구분되지 않는다.
- 근거 부족 답변도 카드로 올릴 수 있어 보인다.
